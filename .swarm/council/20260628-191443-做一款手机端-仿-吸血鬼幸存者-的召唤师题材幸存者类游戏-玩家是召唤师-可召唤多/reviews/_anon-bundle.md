# 召唤师幸存者 H5 — 方案提案（Kimi）

## 一句话方案

用 **Phaser 3 + TypeScript + Canvas/WebGL 2D 渲染** 开发一款竖屏单手操作的召唤师幸存者 H5，玩家操控召唤师走位，召唤 3–5 种自动作战的宠物，通过宠物元素属性连携与局内升级三选一成长，在持续增长的中等规模怪潮中生存计时；核心循环优先于内容量，几周内产出可上手 Demo。

---

## 推荐方案

### 技术栈与框架选择

| 层级 | 选型 | 理由 |
|---|---|---|
| 游戏引擎 | **Phaser 3 (CE 3.80+)** | 2D 生态最成熟、TypeScript 定义完善、AI 训练资料极多；自带 Arcade 物理、对象池、粒子、相机、输入系统；支持 WebGL/Canvas 回退，中端手机友好。 |
| 语言 | **TypeScript** | 主流、类型安全、顶级 AI 生成质量高、便于持续迭代。 |
| 构建 | Vite + esbuild | 秒级热更新、打包小、现代浏览器原生 ESM。 |
| 渲染 | Phaser 自带 WebGL，复杂特效不足时叠加 Canvas 2D 后期（如全屏震屏、连携光环）。 | 避免自研渲染器，利用引擎 Batch Draw。 |
| 资源 | 几何/程序化美术为主，AI 生成（Stable Diffusion / Ideogram）为辅；音效用 jsfxr / sfxr.me。 | 零成本、迭代快、风格统一。 |

### 整体代码架构

采用 **轻量 ECS（Entity-Component-System）风格** 组织逻辑，但不引入重型 ECS 框架，避免 AI 与调试负担。

```
src/
  main.ts                # Phaser Game 配置、场景注册
  scenes/
    BootScene.ts          # 资源加载、localStorage 读档
    MenuScene.ts          # 开始/继续/设置
    GameScene.ts          # 核心战斗场景
    UpgradeScene.ts      # 升级三选一（弹层或暂停浮层）
    ResultScene.ts        # 结算
  systems/
    SpawnSystem.ts        # 波次/难度曲线生成敌人
    MovementSystem.ts     # 召唤师移动、宠物跟随/索敌
    CombatSystem.ts       # 攻击、命中、伤害结算
    SynergySystem.ts      # 连携触发与计算
    UpgradeSystem.ts      # 经验、升级、池子
    SaveSystem.ts         # localStorage 读写
  entities/
    Summoner.ts, Pet.ts, Enemy.ts, Projectile.ts, ExpGem.ts, Pickup.ts
  components/
    Transform, Health, Velocity, Attack, PetData, EnemyData, ElementType
  ui/
    HUD.ts, VirtualJoystick.ts, UpgradeCard.ts, ResultPanel.ts
  utils/
    SpatialGrid.ts, ObjectPool.ts, MathUtils.ts
```

**关键模块说明：**

- **主循环**：Phaser 的 `update(time, delta)` 驱动所有 System；System 按固定顺序执行：输入→移动→碰撞→索敌→战斗→连携→清理→渲染。物理用 Arcade 做粗略碰撞，自定义 SpatialGrid 做范围查询（索敌、AOE）。
- **空间划分**：屏幕内怪物峰值目标约 80–120 只，使用 **网格大小 ≈ 128px 的 SpatialGrid** 替代 O(n²) 距离检测；宠物每 0.2s 一次兴趣索敌（throttle），避免每帧开销。
- **对象池**：`EnemyPool`、`ProjectilePool`、`ParticlePool`、`ExpGemPool`，敌人死亡后回收 Sprite/Graphics，避免 GC 顿卡。
- **碰撞**：Arcade Physics 处理“子弹-敌人”与“敌人-召唤师”碰撞；宠物对敌人伤害用自定义范围检测，不依赖物理体，减少刚体数量。

### 3–5 种原型宠物设计

每只宠物用 **固定主色调 + 发光颜色 + 几何造型** 保证战场上瞬间辨识，并在 HUD 角落用小头像+元素图标显示状态。

| 宠物 | 元素 | 攻击方式 | 定位 | 视觉辨识 |
|---|---|---|---|---|
| **焰灵狐** | 火 | 短程火焰喷射（扇形 AOE，持续伤害） | 前排清杂、点燃 | 橙红菱形狐狸，尾部拖焰粒子 |
| **霜晶蝶** | 冰 | 中距离追踪冰晶，命中减速 | 控场、削速 | 淡蓝蝴蝶，翅膀有冰晶碎片光晕 |
| **雷羽鹰** | 雷 | 高攻速单体闪电链（弹射 2–3 个目标） | 点杀精英、连锁 | 紫黄鹰形，周身间歇闪电 |
| **藤蔓灵** | 自然 | 地面荆棘，持续存在 3s，踩中流血 | 区域封锁、DOT | 绿色藤蔓团，触手摆动 |
| **暗蝠** | 暗 | 低空环绕召唤师，冲撞穿透并吸血反哺 | 续航、近战护盾 | 黑紫蝠群剪影，死亡时飘散 |

每只宠物 2 级解锁“觉醒形态”：焰灵狐喷射变宽、霜晶蝶减速变冻结、雷羽鹰弹射次数+1 等，视觉上加发光环与体积微增。

### 宠物配合/连携机制

核心机制：**元素反应**（简单规则、高视觉反馈）。

- 宠物自带元素标记：火 / 冰 / 雷 / 自然 / 暗。
- 敌人在 2 秒内连续受到两种元素伤害会触发“连携反应”，冷却 1.5 秒/敌人。

| 连携组合 | 效果 | 实现方式 | 视觉反馈 |
|---|---|---|---|
| **火 + 冰 = 蒸汽爆发** | 造成额外 50% 伤害 + 小范围 AOE | 敌人身上同时有火、冰 debuff 时触发一次即时伤害并清 debuff | 敌人位置出现白色蒸汽云+向外扩散冲击波 |
| **雷 + 自然 = 感电荆棘** | 对周围敌人造成连锁闪电并定身 0.8s | 雷命中带“荆棘流血”敌人时，以该敌人为中心释放 4 道闪电链 | 地面绿色电网闪烁，被连锁敌人头顶短暂禁锢图标 |
| **火 + 雷 = 过载** | 目标爆燃，对附近敌人造成溅射 | 可选隐藏组合，Demo 可只实现前两个，第三个作为后续扩展 |

**实现细节：**

- 每个 `Enemy` 维护 `elementMarks: Map<ElementType, {time, source}>`，记录最近 2s 内受到的元素。
- `SynergySystem` 每帧遍历带标记的敌人，检查组合，触发后进入 `synergyCooldown` 状态。
- 连携触发时调用 `CombatSystem` 计算伤害，生成 `EffectEvent` 给 `VFXSystem` 播放，并弹出 **大字号飘字**（如“蒸汽爆发！”）在敌人上方。
- 当玩家同时携带“火+冰”两只宠物时，HUD 中两只宠物头像之间出现动态连线，表示连携可用，强化 build 感知。

### 成长升级系统

#### 局内升级（吸血鬼幸存者式三选一）

召唤师拾取经验宝石升级，每次升级弹出 **三张升级卡**（竖屏底部上滑卡片，或居中弹层）：

| 卡类型 | 示例 | 规则 |
|---|---|---|
| 新宠物 | 召唤“焰灵狐” | 未拥有的宠物优先出现；最多同时携带 5 只 |
| 宠物升阶 | 焰灵狐 → Lv2 / Lv3 / 满级 | 随机从已拥有宠物中抽；满级后不再出现 |
| 召唤师强化 | 移动速度 +10% / 拾取范围 +15% / 冷却缩减 | 通用全局增益 |
| 特殊连携 | 解锁“感电荆棘”组合 | 当拥有对应两种元素宠物时出现 |

- **宠物等级**：共 5 级，每升一级提升攻速/伤害/范围；3 级与 5 级为质变点（如解锁新攻击方式、增加目标数）。
- **升级池**：用加权随机 + 去重保证每局差异；卡面展示图标、名称、效果预览、稀有度颜色。

#### 局外永久成长（雏形）

- 仅保留“符文盘”或“天赋树”的最小版本：
  - 初始生命 +1
  - 初始移动速度 +5%
  - 经验获取 +5%
  - 开局可锁定一只首发宠物
- 用击败 Boss/波次奖励的“灵尘”购买，localStorage 存档。Demo 阶段 5–8 个节点即可，不做复杂网络。

### 移动端操作方案与 UI 布局

#### 操作

- **虚拟摇杆**：左下角半透明圆形摇杆，跟随手指按下位置动态生成（不固定圆心），支持点按即走、拖拽持续移动，松手召唤师停止。
- **自动攻击**：宠物完全自动；召唤师无普攻按钮，突出“指挥而非亲砍”。
- **交互按钮**：右下角仅在需要时显示“拾取/确认”或“跳过选卡”的半透浮钮。

#### 布局（竖屏 9:19.5 参考）

| 区域 | 内容 | 具体排布 |
|---|---|---|
| 顶部 | 计时、当前波次、击杀数、暂停 | 左上计时（最大字号），右上暂停（🔘）+ 设置 |
| 召唤师头顶 | 召唤师血条（细绿条）+ 经验条（细蓝条） | 跟随角色，但 HUD 同时提供稳定血球 |
| 右上角 | 宠物编队小头像（圆形，带元素色环与等级点） | 点击可查看当前宠物等级与技能说明 |
| 底部 | 虚拟摇杆（左）+ 升级卡时居中弹层 | 选卡时背景变暗 70%，三张卡横向滑动 |
| 中央弹层 | 升级三选一 / 结算 / 暂停 | 大按钮、少文字、拇指可单手点击 |
| 屏幕反馈 | 受击红边、连携大字、经验宝石高亮吸向玩家 | 所有弹字从触发点向上飘 1s 后淡出 |

**手感细节：**

- 摇杆死区 12%，边缘限制在屏幕左下 35% 区域；召唤师移动速度 120–180 px/s，保持顺畅但不过快。
- 宠物会 **适度预判** 召唤师移动方向，避免站桩感；当召唤师静止时，宠物围绕召唤师做轻微环绕。
- 经验宝石在靠近召唤师时会被“磁吸”飞入，产生小粒子轨迹，强化爽感。
- 结算界面：显示存活时间、击杀数、获得灵尘、本局最高宠物等级，并给出“再来一局 / 返回主菜单”两个大按钮。

### 性能方案（大量敌人保帧）

目标：中端手机（骁龙 7 系 / 天玑 900）60 只以上敌人 + 50 个投射物保持 **30–60fps**。

1. **对象池**：所有敌人、子弹、宝石、粒子全部池化，避免运行时创建/销毁 DisplayObject。
2. **SpatialGrid**：屏幕按 128×128 分格，敌人只更新所在格与相邻格，宠物索敌从最近格开始向外搜索。
3. **战斗节流**：
   - 宠物攻击冷却 ≥ 0.2s；
   - 敌人伤害结算每 0.1s 批量处理一次（非每帧）；
   - 连携判定只对有元素标记的敌人执行，且每敌人有冷却。
4. **渲染优化**：
   - 敌人统一用 `Phaser.GameObjects.Graphics` 或纹理图集，批量渲染；
   - 开启 Phaser 的 `roundPixels: true` 与 `mipmap: false`；
   - 屏幕外实体停止更新与渲染（culling）。
5. **特效分级**：低端机关闭粒子/阴影，仅保留简单色块与闪白。
6. **帧率自适应**：当 fps < 30 连续 2 秒，自动减少同屏敌人上限与粒子数量。

## 关键决策与取舍

| 选择 X | 放弃 Y | 原因 |
|---|---|---|
| **Phaser 3** | PixiJS 裸写 | Phaser 自带物理、场景、输入、对象池，AI 生成更可靠；PixiJS 更灵活但需自搭循环与碰撞，原型期 overhead 大。 |
| **Arcade Physics + 自定义 SpatialGrid** | Matter.js / Box2D | 吸幸类游戏无需真实物理，简单 AABB 即可；重型物理徒增性能风险。 |
| **轻量自研 ECS 风格** | 引入 bitECS 等库 | 避免团队/AI 学习成本，保持代码可读性；后期若内容爆炸再迁移。 |
| **元素反应式连携** | 复杂阵型/手动技能连携 | 自动触发不增加操作负担，符合“召唤师指挥”定位；视觉反馈强，规则易理解。 |
| **竖屏单手摇杆** | 横屏双摇杆 | 移动浏览器竖屏更自然，单手通勤可玩；双摇杆会要求玩家同时操作攻击，弱化“宠物自动作战”的特色。 |
| **几何+程序美术占位** | 购买/外包美术 | 成本 ≈ 0，迭代快；AI 生成仅用于 Splash 与宣传图，游戏内靠色块+粒子即可辨认。 |
| **localStorage 存档** | 后端存档 | 零服务器成本，符合 H5 原型定位；后续可扩展为云存档。 |

## 步骤 / 任务分解

按 **3 周 Demo** 规划，每周 5 个工作日强度。

### W1：核心循环与可移动召唤师

- 搭建 Phaser 3 + Vite + TypeScript 工程；配置竖屏、摇杆、输入。
- 召唤师移动、镜头跟随、血条/经验条 HUD。
- 简单敌人（几何图形）生成、移动、碰撞、死亡掉落经验宝石。
- 经验拾取 + 升级弹层框架（仅三选一占位）。
- **W1 交付物**：能跑、能移动、能杀怪、能升级（数值为占位）的 H5 页面。

### W2：宠物系统、连携与完整成长

- 实现 3 只核心宠物：焰灵狐、霜晶蝶、雷羽鹰（攻击、索敌、升级）。
- 宠物跟随/编队逻辑、对象池、宠物 HUD 头像。
- 实现“火+冰”蒸汽爆发连携，包含视觉反馈与伤害结算。
- 升级三选一池子：新宠物、宠物升阶、召唤师强化。
- 局外天赋雏形（灵尘、节点、localStorage 存档）。
- **W2 交付物**：完整召唤师+宠物战斗循环，可感受连携与成长。

### W3：内容填充、性能与移动端打磨

- 追加 2 只宠物：藤蔓灵、暗蝠；实现“雷+自然”感电荆棘连携。
- 波次/难度曲线：敌人生成速率、血量、速度随时间递增；加入小精英怪。
- 性能优化：对象池、SpatialGrid、节流、低配降级。
- UI 打磨：选卡滑动、结算界面、连携大字、受击红边、磁吸经验特效。
- 多端测试（Chrome DevTools 手机模拟、至少 1 台真实中端 Android 手机）。
- **W3 交付物**：可上手玩的 Demo，含主菜单、战斗、升级、结算、存档。

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| **中端手机同屏怪物多掉帧** | 高 | 对象池+SpatialGrid+节流三重；帧率自适应；Demo 期同屏上限先设为 80 只。 |
| **连携机制太复杂或反馈弱** | 中高 | 只做 2 种反应，触发即清 debuff，保证“触发-反馈”即时；用大字+粒子+震屏强化。 |
| **宠物 AI 索敌卡顿** | 中高 | 宠物索敌 0.2s 一次；SpatialGrid 加速；优先攻击最近目标。 |
| **虚拟摇杆手感差** | 中 | 动态摇杆、死区可调、边缘限制；W1 就进行真机测试并快速迭代。 |
| **AI 生成代码难以持续维护** | 中 | 类型化架构、清晰模块边界、每个 System 单一职责；保留 ESLint + Prettier 规范。 |
| **美术风格不统一** | 低 | 全部用几何+发光色块，Splash 与图标才用 AI 生成；统一调色板（火橙/冰蓝/雷紫/自然绿/暗黑）。 |
| **Phaser 3 在部分 WebView 兼容性** | 低 | 开启 Canvas 回退；打包后真机测试微信/QQ/系统浏览器。 |

## 验收标准（done-when，可检验）

1. 在中端 Android 手机浏览器（或 Chrome DevTools 6x CPU 降速）中，**同屏 60 只敌人 + 40 个投射物时帧率 ≥ 30fps**。
2. 召唤师通过左下角虚拟摇杆可单手流畅移动，摇杆响应延迟 < 100ms。
3. 玩家至少可同时携带 **3 只不同宠物** 并自动作战，且能在战场上凭颜色/形状快速区分。
4. **“火+冰”连携可被稳定触发**，屏幕出现“蒸汽爆发”飘字与冲击波特效，并造成可见 AOE。
5. 升级三选一完整：每级弹出 3 张卡，可选“新宠物 / 宠物升阶 / 召唤师强化”，选择后即时生效。
6. 一局可存活 **至少 5 分钟**，波次/难度曲线明显递增，结算界面展示存活时间、击杀数、灵尘。
7. 局外成长存档可读写 localStorage，重启浏览器后天赋/灵尘保留。
8. 核心 UI 在 375×812 与 414×896 两种竖屏分辨率下无重叠、无截断。

## 推荐方案

### 技术栈与渲染

- **框架**：Phaser 3（最新 3.80+）。理由：
  1. AI 训练语料最充足的 2D 框架，生成的代码命中率最高、最容易持续迭代——这是本课题的硬约束之一。
  2. 自带场景/输入/音频/纹理/对象池/粒子，省掉大量手写脚手架，AI 也能直接用其惯用 API。
  3. 自动 WebGL→Canvas 降级，中端机也有兜底。
- **语言**：TypeScript（strict）。类型边界清晰，AI 跨次迭代不容易把模块搞乱。
- **渲染**：Phaser 默认 WebGL 批量渲染；精灵用一张打包图集（TexturePacker 免费 / AI 生成后手动合图），减少 draw call。粒子用 Phaser ParticleEmitter（内置批量）。
- **不用** Arcade Physics 的逐对 collider 去跑几百个敌人（性能灾难），改用**自写空间哈希网格**做碰撞，Arcade 只留给召唤师与少量拾取物的粗筛。

### 代码架构（关键模块清单）

采用"Phaser 管场景与渲染 + 自写 ECS 管逻辑"的混合。ECS 不引第三方库（避免冷门、AI 不熟），用极简自实现：

```
src/
  main.ts                 # Phaser game config、场景注册、缩放策略
  scenes/
    BootScene.ts          # 资源预加载
    GameScene.ts          # 主战场，持有 world 与主循环 update
    UIScene.ts            # 叠加层：摇杆/HUD/升级卡/结算（独立 scene，不被游戏缩放拉伸）
  ecs/
    World.ts              # 实体容器 + 系统调度 + 空间网格
    Entity.ts             # id + component map
    components/           # Transform/Health/Damage/Enemy/Pet/ElementTag/Timer...
    systems/
      SpawnSystem.ts      # 怪潮刷新 + 难度曲线
      PetAISystem.ts      # 宠物寻敌 + 攻击行为（按 PetArchetype 策略）
      EnemySystem.ts      # 敌人朝召唤师移动
      CombatSystem.ts     # 伤害结算（节流 + 元素反应）
      ElementSystem.ts    # 元素标签叠加 + 反应触发
      PickupSystem.ts     # 经验宝石吸附 + 升级
      MovementSystem.ts   # 召唤师/敌人位移 + 摇杆输入
  game/
    PetArchetype.ts       # 5 种宠物原型定义（数据驱动）
    EnemyFactory.ts
    UpgradePool.ts        # 升级三选一卡池
    MetaProgression.ts    # 局外成长 + localStorage
    Balance.ts            # 难度曲线/数值集中表
  input/
    VirtualJoystick.ts    # 触屏虚拟摇杆（插件 phaser3-rex-plugins 的 virtual-joystick，AI 熟）
ui/
  Hud.ts / UpgradeCardView.ts / ResultView.ts
```

**对象池**：`EnemyPool`、`ProjectilePool`、`ParticlePool`、`GemPool`。预分配，复用，禁用 `new`/`destroy` 在热路径。

**空间哈希网格**：cell = 64px。每帧重建（只对移动实体），碰撞查询 O(邻居) 而非 O(n²)。150+ 敌人时可稳 60fps。

**主循环节流**：
- 伤害结算每 100ms tick 一次（而非每帧），表现层用插值。
- 敌人寻路每 200ms 重算一次目标方向（不需要每帧）。
- 粒子用 Phaser 内置批量，限制 emitter 上限。

### 5 种原型宠物（数据驱动，`PetArchetype.ts` 一张表）

| 宠物 | 攻击方式 | 定位 | 元素 |
|---|---|---|---|
| 剑齿狼 | 冲向最近敌人挥砍（近战单体高伤） | 近战冲锋 | 无（物理） |
| 焰魔 | 投火球，命中爆炸 AOE | 远程 AOE | 火 |
| 冰晶蝶 | 周身光环持续给范围内敌人挂"霜冻"减速 | 控场辅助 | 冰 |
| 雷霆鸟 | 闪电链，在 3-4 个邻近敌人间弹射 | 远程连锁 | 雷 |
| 治愈花 | 周期性给召唤师回血 + 短暂护盾 | 生存辅助 | 无（辅助） |

每种宠物有独立等级（1→5），升级提升伤害/攻速/范围；玩家最多同时带 5 只宠物。

### 连携/配合机制：元素反应（核心特色）

敌人身上可叠加**元素标签**（火/冰/雷，带层数与持续时间）。当两个不同元素同时存在时触发一次**反应**，清空对应标签并进入该敌人 1.5s 反应冷却（防刷屏）。

具体 2 个反应（够 demo，可扩展）：

1. **融化（火 + 冰）**：焰魔的火打中已被冰晶蝶挂冰的敌人 → 立即爆一次 AOE 伤害（半径 60px），伤害 = 火伤 × 2.0，并给周围敌人重新挂少量火。
   - 触发判定：`CombatSystem` 结算火伤时查 `ElementComponent`，若冰层数 ≥ 1 → 触发融化、清冰、生 AOE。
2. **超导（冰 + 雷）**：雷霆鸟的电打中挂冰敌人 → 释放一次冲击波，造成雷伤 × 1.5 并对周围敌人施加 1.5s 减速。
   - 触发判定：同理在雷伤结算时查冰层。

这让玩家有"先放冰晶蝶铺冰、再用焰魔/雷霆鸟引爆"的指挥节奏感，而不是各打各的。**治愈花/剑齿狼不参与反应**，做物理与生存兜底，降低反应系统的认知负担。

实现要点：`ElementSystem` 单独跑标签衰减与冷却；反应在 `CombatSystem` 内联触发（不另开系统，省同步成本）。

### 成长升级系统

**局内**：
- 杀敌掉经验宝石（吸附式拾取，吸幸经典）。召唤师经验条满 → 弹**升级三选一**卡（`UpgradePool`）。
- 卡池三类混合出现：
  - **强化现有宠物**：如"焰魔：伤害 +20% / 攻速 +15% / 范围 +30%"
  - **解锁新宠物**：召唤一只未拥有的宠物（场上未满 5 只时）
  - **通用被动**：如"召唤师移速 +10% / 拾取范围 +50% / 经验加成 +15% / 反应伤害 +25%"
- 宠物等级随其相关卡被选而升（1→5 满级），满级后该宠的强化卡不再出现。

**局外永久成长（雏形）**：
- 死亡后结算"魂晶"（按存活时间/击杀数给）。`MetaProgression` 用 localStorage 存。
- 可解锁：新宠物原型初始可用、召唤师基础移速/血量上限、初始带一只宠物开局、反应伤害全局加成。量小，主要证明 meta 循环存在即可。

### 移动端操作与 UI 布局

- **操作**：左下虚拟摇杆（拖拽）控制召唤师移动；宠物全自动，无需攻击键。**单手可玩**。
- **UI（独立 UIScene，固定不缩放）**：
  - 顶部：计时器 / 等级条 / 击杀数 / 当前波次
  - 右下：5 个宠物图标格（显示头像 + 等级数字，灰显=未召唤）
  - 升级时：底部弹 3 张大卡，点击选择（点空白不关闭，强制选）
  - 结算：存活时间 / 击杀 / 获得魂晶 / 再来一局 / 返回菜单
- **缩放**：`Scale.FIT` + 自适应竖屏，安全区适配刘海。横屏可选但 demo 以竖屏为主（更符合"手机随手玩"）。

### 性能方案（大量敌人时保帧率）

1. **对象池**：敌人/弹道/宝石/粒子全部池化，热路径零分配。
2. **空间哈希网格**：cell 64px，只对移动实体重建，碰撞与"宠物寻最近敌"都走网格。
3. **批量渲染**：单张图集 + Phaser 自动 batching；同纹理敌人共享 batch。
4. **伤害结算节流**：100ms tick；寻路 200ms tick；不在每帧做 O(n) 遍历的"最近敌搜索"（缓存网格结果）。
5. **同屏上限**：敌人软上限 200，超出后停止新刷只加血量（吸幸同款处理）。
6. **粒子限流**：单 emitter 上限，死亡爆裂用少量大粒子而非海量小粒子。
7. **避免 GC**：复用 Vector/临时对象，禁用热路径字符串拼接（用数字 id）。
8. **目标**：中端安卓 Chrome / iOS Safari，150 敌人时 ≥30fps，100 敌人时 ≥60fps。

## 关键决策与取舍

- **选 Phaser 3 而非 PixiJS+裸写**：PixiJS 渲染更快，但缺场景/输入/音频脚手架，AI 要补大量胶水代码，几周内更易翻车。Phaser 性能对本课题够用（上限 200 敌人 + 节流），换取 AI 迭代速度与脚手架成熟度。**取舍点：性能换可迭代性，值得。**
- **选自写轻量 ECS 而非 bitECS 等**：第三方 ECS 库 AI 不熟、冷门、调试难。自写 60 行的 ECS + component map 足够，且 AI 能稳定生成。
- **选空间哈希网格而非 Arcade collider**：Arcade 逐对检测在 200 敌人时会崩帧。自写网格是本课题的性能命门，必须做。
- **选元素反应做连携而非"宠物合体/合招"**：反应是状态叠加型，实现简单（标签 + 结算时判定），玩家有指挥感又不用复杂 AI 编队逻辑。合体/合招 demo 内做不起且难调。
- **选竖屏 FIT 而非横屏**：竖屏单手摇杆更顺手，符合"手机随手玩"目标；横屏需双拇指，门槛高。
- **选 5 宠物 2 反应**：覆盖近战/远程/AOE/控场/辅助全定位，反应只做 2 个够展示机制，避免内容膨胀拖慢 demo。

选择 Phaser 3 的原因：

1. **AI 生成代码友好**：Phaser 3、TypeScript、Vite 的训练语料充足，AI 能稳定生成场景、输入、物理、动画、资源加载、UI 等常见结构。
2. **移动 H5 成熟**：Phaser 对触摸输入、缩放、音频解锁、WebGL 批渲染和浏览器兼容有成熟方案。
3. **原型速度快**：内置 Scene、GameObjects、Tweens、Camera、Input、Loader，几周内做可玩 Demo 比 PixiJS + 自建游戏框架更稳。
4. **性能可控**：不使用 Arcade Physics 做全部单位碰撞，核心战斗采用自写轻量数据层 + 空间网格，Phaser 主要负责渲染和输入，避免大量物理体拖垮手机。

### 整体代码架构

采用 **“轻量 ECS 思路的数据驱动架构”，不引入复杂 ECS 库**。原因是 Demo 期需要清晰、低依赖、AI 易维护：实体用数字 id 和 TypedArray/普通数组存核心状态，系统按固定顺序更新。

关键主循环：

1. `InputSystem`：读取虚拟摇杆向量，更新召唤师移动意图。
2. `SpawnSystem`：按计时器、难度曲线和屏幕边缘刷新怪物。
3. `PetAISystem`：宠物寻敌、跟随主人、施法冷却、阵型调整。
4. `EnemySystem`：敌人朝召唤师推进，处理简单避让或分层速度扰动。
5. `ProjectileSystem`：更新投射物、命中检测、生命周期。
6. `CombatSystem`：处理伤害、状态、连携触发、死亡。
7. `ExpSystem`：经验球掉落、吸附、升级。
8. `RenderSyncSystem`：把数据层位置、血量、状态同步到 Phaser GameObjects。
9. `UISystem`：HUD、暂停、升级三选一、结算。

关键模块清单：

- `GameScene`：主战斗场景，持有系统调度与时间轴。
- `BootScene / PreloadScene`：初始化、加载资源。
- `UIScene`：HUD、选卡、结算浮层，和战斗 Scene 解耦。
- `EntityManager`：管理玩家、宠物、敌人、投射物、经验球 id。
- `ObjectPool<T>`：敌人、子弹、伤害数字、经验球、特效复用。
- `SpatialGrid`：固定网格空间划分，用于敌人查询、AOE、投射物命中。
- `PetConfig / EnemyConfig / UpgradeConfig`：所有数值配置化，便于 AI 后续扩内容。
- `SaveService`：localStorage 读写局外成长、最高存活时间、设置。
- `BalanceService`：难度曲线、刷新权重、升级池权重。

碰撞与空间划分：

- 地图按固定格子切分，例如 `128px × 128px`。
- 敌人每帧写入 `SpatialGrid`，宠物攻击只查询目标附近格子，不全局遍历。
- 近战、AOE、吸经验都用圆形距离平方判断。
- 子弹命中使用“查询当前格和邻格 + 半径判断”，不使用复杂物理引擎。

对象池：

- 敌人、投射物、经验球、短命特效全部池化。
- 敌人死亡只隐藏并回收，不销毁 GameObject。
- 伤害数字可默认关闭或限频显示，低端机可自动降级。

### 3-5 种原型宠物设计

建议首版做 5 种，但 W1 先完成 3 种，W2 补齐：

1. **狼灵**
   - 定位：近战单体 / 前排。
   - 攻击方式：自动扑向最近敌人，短距离撕咬，命中后返回召唤师附近或继续追击。
   - 特色：对“被标记”目标造成额外伤害，承担基础割草手感。

2. **火鸦**
   - 定位：远程穿透 / 点燃。
   - 攻击方式：绕召唤师盘旋，定时发射火羽，火羽可穿透 2-4 个敌人。
   - 特色：给敌人叠加 `Burn`，持续掉血，适合清线。

3. **藤妖**
   - 定位：控制 / AOE 辅助。
   - 攻击方式：在敌群脚下生成藤蔓圈，短暂减速并每 0.5 秒造成伤害。
   - 特色：把敌人聚成低速团，给火鸦和雷灵创造输出窗口。

4. **雷灵**
   - 定位：跳跃链 / 爆发。
   - 攻击方式：对最近敌人释放闪电，随后跳到附近 2-5 个目标。
   - 特色：对处于 `Wet / Rooted / Marked` 的目标增加跳跃次数或暴击。

5. **守护龟**
   - 定位：防御 / 回复 / 容错。
   - 攻击方式：低频震地小 AOE，主要提供护盾光环。
   - 特色：每隔一段时间给召唤师加护盾，升级后可嘲讽小范围敌人或减伤。

### 宠物配合 / 连携机制

推荐核心机制：**“召唤印记 + 元素反应”双层连携，但首版只做 2 个反应，避免复杂失控。**

通用实现：

- 每个敌人有 `statusFlags` 和 `statusStacks`，例如 `Marked`、`Burn`、`Rooted`。
- 每次宠物命中时调用 `ComboSystem.onHit(attackerPetType, targetEnemyId, hitPayload)`。
- `ComboSystem` 根据目标状态和攻击类型触发额外效果，并设置内置冷却，防止同一敌人每帧无限触发。
- 连携伤害使用独立系数，受宠物等级、召唤师天赋和局内升级影响。

具体连携例子 1：**狼灵 + 火鸦：猎印爆燃**

- 触发：
  - 狼灵攻击有 35% 概率给敌人挂 `Marked`，持续 4 秒。
  - 火鸦火羽命中 `Marked` 敌人时触发“爆燃”。
- 计算：
  - 爆燃造成 `火鸦伤害 × 0.8 + 狼灵等级 × 4` 的小范围 AOE。
  - AOE 半径 `80px`，最多影响 8 个敌人。
  - 每个被标记敌人每 1.2 秒最多触发一次，避免性能和数值爆炸。
- 体验：
  - 玩家感觉不是自己攻击，而是“狼灵锁定猎物，火鸦引爆猎物”，突出指挥召唤主题。

具体连携例子 2：**藤妖 + 雷灵：缚雷连锁**

- 触发：
  - 藤妖藤蔓圈内敌人获得 `Rooted` 或 `Slowed`。
  - 雷灵命中 `Rooted/Slowed` 敌人时，链式闪电额外跳跃 2 次。
- 计算：
  - 基础跳跃 `3` 次；若首目标有 `Rooted/Slowed`，变为 `5` 次。
  - 每次跳跃伤害衰减 15%。
  - 每次跳跃只查询目标所在格周边 `160px` 内最近敌人。
- 体验：
  - 藤妖把怪潮“定住成团”，雷灵负责“清屏跳电”，形成非常直观的组合构筑。

### 成长升级系统

局内基础循环：

1. 怪物死亡掉经验球。
2. 召唤师靠近吸取，或升级后获得更大吸附半径。
3. 经验条满时暂停战斗，弹出“升级三选一”。
4. 玩家选择一张卡，立即强化宠物、召唤师或连携。
5. 继续战斗，怪潮按时间增强。

升级三选一卡池示例：

- 宠物获得类：
  - `召唤狼灵`：若未拥有则新增狼灵，已拥有则狼灵等级 +1。
  - `召唤火鸦`：新增或升级火鸦。
  - `召唤藤妖`：新增或升级藤妖。
  - `召唤雷灵`：新增或升级雷灵。
  - `召唤守护龟`：新增或升级守护龟。
- 宠物强化类：
  - `兽群号令`：所有宠物攻击速度 +12%。
  - `灵魂契约`：宠物伤害 +15%，召唤师最大生命 -5%，做风险收益。
  - `迅捷召回`：宠物离主人过远时回位速度 +30%。
- 连携强化类：
  - `猎印扩散`：猎印爆燃命中 3 个以上敌人时，有 25% 概率给附近敌人扩散 `Marked`。
  - `缚雷增幅`：雷灵命中减速敌人时额外跳跃 +1。
  - `元素共鸣`：触发任意连携后，所有宠物冷却缩短 0.2 秒，内置冷却 3 秒。
- 召唤师生存类：
  - `灵能护盾`：每 20 秒获得一层护盾。
  - `轻身术`：移动速度 +8%。
  - `拾魂范围`：经验吸附范围 +25%。

宠物等级提升：

- 每种宠物等级 1-8。
- 升级方向固定但可配置，例如：
  - 等级提升增加基础伤害、冷却、范围、投射物数量。
  - 关键等级解锁质变：火鸦 4 级穿透 +1，藤妖 5 级藤蔓圈 +1，狼灵 6 级标记概率提升。
- 卡池权重优先给已拥有宠物，保证玩家能形成构筑，而不是随机太散。

可选局外成长雏形：

- 只做轻量 localStorage，不做账号和服务器。
- 结算后获得“魂石”，可点 3 个永久升级：
  - 初始生命 +5%。
  - 初始移动速度 +3%。
  - 初始随机宠物等级 +1，最多 1 级。
- 局外成长必须弱，不破坏 Demo 平衡，主要用于展示可扩展方向。

### 移动端操作方案与 UI 布局

操作：

- 竖屏优先，推荐设计分辨率 `720 × 1280`，用 Phaser Scale `FIT + CENTER_BOTH`。
- 左下角虚拟摇杆：
  - 手指按下出现摇杆底盘。
  - 拖拽控制移动方向和速度。
  - 松手停止。
- 右下角保留一个“召唤爆发 / 集火指令”按钮作为可选增强：
  - 首版可以做成“所有宠物优先攻击最近精英 / Boss，持续 4 秒”。
  - 若赶工，可只展示按钮但不做复杂技能；核心仍是自动战斗。

HUD：

- 顶部中间：存活时间、当前难度阶段。
- 顶部左侧：生命 / 护盾。
- 顶部右侧：暂停按钮、性能降级提示。
- 底部：经验条。
- 屏幕右侧小图标列：当前宠物和等级。
- 升级弹窗：三张大卡片纵向或横向排列，手机上卡片要足够大，显示名称、图标、效果和“新召唤 / 升级 / 连携”标签。
- 结算 UI：存活时间、击杀数、最高等级、触发连携次数、获得魂石、重新开始按钮。

### 性能方案

目标：中端手机浏览器在 150-300 个敌人、数十个投射物和少量特效时保持接近 30-60 FPS，优先保证可玩。

具体策略：

1. **不把每个敌人做成完整 Phaser 物理体**
   - 敌人移动和碰撞由自写数据系统处理。
   - Phaser 只渲染 sprite 或简单形状。

2. **固定空间网格**
   - 敌人每帧按格子登记。
   - 宠物寻敌、AOE、子弹命中只查附近格。
   - 避免 `宠物 × 敌人` 或 `子弹 × 敌人` 全量遍历。

3. **对象池**
   - 敌人、子弹、经验球、爆炸特效、伤害数字全部复用。
   - 每波刷新从池中取对象，死亡时回池。

4. **伤害结算节流**
   - AOE 地面效果每 `200ms` 或 `500ms` tick 一次，不每帧伤害。
   - 连携触发有 per-target cooldown。
   - DOT 每 `500ms` tick。

5. **批量渲染和素材控制**
   - 同类敌人使用同一 texture / atlas。
   - 几何占位阶段使用少量 texture，不频繁动态绘图。
   - 关闭昂贵滤镜、动态光照和复杂粒子。

6. **敌人行为简化**
   - 普通敌人只朝召唤师移动，加少量随机速度偏移。
   - 不做复杂寻路。
   - 敌人之间不做真实碰撞，只做极简分离或干脆允许重叠，用视觉分层解决。

7. **动态降级**
   - FPS 连续 3 秒低于 28 时：
     - 降低伤害数字数量。
     - 降低粒子数量。
     - 降低最大同时敌人软上限。
     - 延长部分视觉特效刷新间隔。

## 关键决策与取舍（选 X 而非 Y，因为…）

1. **选 Phaser 3 而非 PixiJS**
   - PixiJS 渲染更纯粹，但需要自建更多游戏层能力；Phaser 对场景、输入、资源、音频、移动适配更完整，几周 Demo 更稳。

2. **选 TypeScript 而非 JavaScript**
   - 宠物、敌人、升级、状态、连携配置会快速膨胀，TypeScript 能减少 AI 迭代时的字段拼错和接口漂移。

3. **选轻量 ECS 思路而非引入完整 ECS 库**
   - 完整 ECS 学习和调试成本高，AI 生成代码也容易过度抽象；用数组 + 系统模块能获得主要性能收益，同时保持可读。

4. **选自动战斗 + 走位 + 构筑，不做主动普攻**
   - 题材核心是召唤师“指挥而非亲砍”，主动攻击会削弱差异化；玩家操作负担也更适合手机。

5. **选 2 个明确连携而非做复杂元素体系**
   - 连携是硬核心，但 Demo 期复杂元素反应容易数值爆炸和难解释；两个强反馈连携足够证明方向。

6. **选 localStorage 轻局外成长而非账号 / 云存档**
   - 符合零成本和范围约束，能展示长期成长雏形，但不引入服务器复杂度。

7. **选竖屏虚拟摇杆而非横屏双摇杆**
   - 竖屏更符合 H5 传播和单手试玩；游戏不需要瞄准摇杆，左手拖拽移动即可。

## 风险与缓解

1. **大量敌人导致手机掉帧**
   - 缓解：从 W1 开始就用对象池和数据层移动；W2 加空间网格；W3 做动态降级。不要先用 Arcade Physics 堆满敌人再重构。

2. **连携机制看不懂**
   - 缓解：只保留 2 个首发连携，配强视觉标签和升级卡描述；结算展示“触发连携次数”，强化玩家认知。

3. **升级卡太随机，构筑不成型**
   - 缓解：卡池权重优先已有宠物；前 3 次升级提高“新召唤宠物”出现概率；同一宠物升级路径稳定。

4. **召唤师本人太弱导致无聊或无力**
   - 缓解：给召唤师足够移动速度、拾取成长、护盾和“集火指令”按钮；玩家虽然不攻击，但通过走位、选卡、指挥影响战局。

5. **AI 生成代码后期难维护**
   - 缓解：强制配置驱动，模块边界清晰，避免全塞进一个 Scene；所有宠物和升级以 `Config` 表声明。

6. **移动浏览器音频和触摸兼容问题**
   - 缓解：首次触摸后解锁音频；所有按钮触摸热区大于 44px；禁止页面滚动；适配安全区。

7. **数值失控，连携秒屏或完全没用**

## 一句话方案
采用 Phaser 3 (WebGL) + TypeScript 构建的高性能 H5 游戏，以轻量 ECS 架构驱动海量实体，主打"宠物全自动战斗 + 元素连携反应 + 三选一卡池"的生存割草体验。

## 推荐方案
**技术栈选择**
*   **框架与渲染**：选用 **Phaser 3** 配合 **TypeScript**。Phaser 3 是当前极其成熟的 H5 2D 游戏框架，默认使用 WebGL 批处理渲染，API 极其丰富且网上 AI 训练数据覆盖率高，AI 生成代码的准确率和可运行率有保障。
*   **语言**：TypeScript 的强类型有助于在持续迭代和增加多种宠物/卡牌时减少低级拼写和类型 Bug，便于 AI 维护项目结构。

**整体代码架构**
*   **核心架构**：基于轻量级的 **ECS (Entity-Component-System)** 思想。
*   **关键模块**：
    *   `GameScene`：主循环、UI 层与游戏层的解耦管理。
    *   `EntityManager` / `PoolManager`：统一管理主角、宠物、怪物、弹幕、掉落物实体的生命周期与对象池分配。
    *   `DirectorSystem` (AI 导演)：根据存活时间控制敌人波次刷新、怪物密度与属性强度。
    *   `CombatSystem`：处理索敌策略、物理交集、伤害结算与元素连携判定。
    *   `UpgradeSystem`：管理经验值拾取、升级逻辑、"三选一"卡池随机抽取与应用。

**3-5 种原型宠物设计**
1.  **火球灵 (Fire Sprite)**：【远程/单体/火】周期性向最近敌人发射火球，单体伤害高。
2.  **毒蘑菇 (Toxic Shroom)**：【AOE/场地/毒】在自身周围散播毒雾，进入区域的敌人减速并持续掉血（附加中毒 Buff）。
3.  **旋风兽 (Gale Beast)**：【近战/生存/风】环绕在召唤师周围快速旋转，造成轻微伤害并击退靠近的敌人，为主角提供安全区。
4.  **雷电鸟 (Thunderbird)**：【远程/弹射/雷】释放闪电链，在密集敌人间弹射，弹射伤害递减，适合清杂兵。

**宠物配合/连携机制**
*   **设计核心**：通过“状态异常(Buff) + 特定元素攻击”触发进阶反应。
*   **具体连携例子**：
    1.  **毒爆 (毒 + 火)**：当敌人身上带有"中毒"状态（毒蘑菇赋予）时，被"火球灵"击中，会引发爆炸，造成目标最大生命值百分比的范围真实伤害，并清除中毒状态。
    2.  **感电扩散 (风 + 雷)**：被"旋风兽"击退的敌人会被短暂挂上"风破"易伤标记，此时若被"雷电鸟"击中，闪电链的弹射范围翻倍，且不再衰减伤害。
*   **实现方式**：在 `CombatSystem` 的伤害结算环节，读取目标实体的 `BuffList`。若受到攻击属性匹配连携条件，则取消普通伤害逻辑，触发特殊效果（抛出 `ComboEvent`，生成新的范围伤害实体）。

**成长升级系统**
*   **局内升级**：
    *   敌人死亡掉落经验宝石，召唤师靠近触发吸附拾取。经验条满后暂停游戏时间，弹出"三选一"卡牌 UI。
    *   **卡池示例**：【解锁】获得新宠物"雷电鸟"；【升级】"火球灵"弹道数量+1；【被动】全体宠物冷却时间-10%；【强化】"毒蘑菇"毒雾范围+20%。
*   **局外成长 (雏形)**：利用 H5 `localStorage` 记录金币，可在主菜单花费金币提升召唤师初始移动速度、经验获取倍率。

**移动端操作方案与 UI 布局**
*   **操作**：全屏任意位置滑动（虚拟摇杆隐藏显示）控制召唤师移动，纯单指操作。宠物全自动运行。
*   **UI 布局**：
    *   顶部：经验条（贯穿全宽）、居中显示存活倒计时。
    *   左上角：玩家血条。右上角：暂停/设置按钮。
    *   底部：小图标排列展示当前已获得的宠物及等级（Max 5级）。
    *   升级时：屏幕中央弹出 3 张竖向排列或并排的大卡片，点击即选。

**性能方案（保障海量怪物不卡顿）**
1.  **对象池 (Object Pool)**：怪物、弹幕、伤害飘字、经验宝石 100% 使用对象池，杜绝 `new` 操作造成的内存抖动与 GC 停顿。
2.  **空间哈希网格 (Spatial Hash Grid)**：将屏幕按大小划分为网格。寻敌逻辑与碰撞检测只遍历相邻网格内的实体，从 $O(N^2)$ 降为 $O(N)$，解决几百只怪物同屏时的 CPU 瓶颈。
3.  **精简物理**：禁用 Phaser 默认的 Arcade 物理碰撞反弹，仅保留简单的 AABB（矩形）重叠检测。允许怪物模型重叠，避免复杂的物理挤压计算。
4.  **计算降频**：每隔 3-5 帧才进行一次索敌和 AI 移动方向更新，不需要 60 帧每秒的高频计算。

## 关键决策与取舍
*   **选择纯渲染框架而非 DOM/前端框架**：放弃用 React/Vue 渲染实体的思路。割草游戏核心在于成百上千对象的坐标高频更新，必须用 Phaser (WebGL) 直接进行底层批处理渲染。
*   **剥离物理引擎，自定义简单交集**：传统的重度物理引擎（涉及动量、挤压）对割草游戏是灾难。我们选择“允许穿模叠加”，只需判断矩形相交即扣血，极大保全了手机端的 CPU 资源。
*   **召唤师零攻击**：为强化“召唤”特色，主角没有任何攻击能力，完全依赖走位和宠物护航，从机制上与传统射击肉鸽区隔。

## 风险与缓解

**主循环 (GameScene.update):**
```
1. VirtualJoystick → Player.velocity
2. Player.update() — 移动 + 经验磁铁
3. PetAISystem.update() — 每个宠物寻敌 → 攻击 → 生成弹幕
4. SynergySystem.update() — 扫描所有敌人的元素标记 → 触发连携
5. SpawnSystem.update() — 按波次曲目表出怪
6. Enemy.update() × N — AI 寻路向玩家移动
7. Phaser Arcade Physics overlap/collide — 弹幕命中检测 (走 SpatialGrid 过滤后)
8. DamageSystem.update() — 结算伤害 + 死亡回收
9. XPSystem.update() — 经验宝石拾取 → 升级触发 → 弹出 UpgradePanel
10. WaveSystem.update() — 计时器 + 难度递增
11. HUD.update() — 刷新 UI
```

### 5种原型宠物设计

| # | 名称 | 攻击方式 | 定位 | 攻击范围 | 攻击间隔 | 基础伤害 | 特殊机制 |
|---|------|---------|------|---------|---------|---------|---------|
| 1 | **岩石傀儡** 🪨 | 近战冲撞，撞击最近敌人 | 前排坦克 | 60px | 2.0s | 15 | 攻击附带小范围击退；每10秒自动嘲讽周围敌人（吸引仇恨） |
| 2 | **火焰精灵** 🔥 | 发射火球，命中产生60px爆炸 | 远程AOE | 250px | 1.2s | 20 | 爆炸溅射伤害 = 直接伤害的50%；给目标上"火标记" |
| 3 | **冰霜之狼** ❄️ | 快速扑咬，三连击 | 近战控制 | 80px | 0.6s (×3) | 8×3 | 每击附加"冰标记"；身上已有冰标记的敌人被再次击中 → 冰冻（定身1秒） |
| 4 | **雷电之眼** ⚡ | 闪电链，弹跳至附近敌人 | 远程连锁 | 200px | 1.5s | 18 | 最多弹跳4次，每次伤害衰减15%；弹跳优先找"有元素标记"的敌人；给目标上"雷标记" |
| 5 | **治愈之灵** 💚 | 每2.5秒对召唤师+血量最低宠物释放治疗 | 辅助治疗 | 150px | 2.5s | 治疗12 | 不攻击；治疗时若目标身上有元素标记，治疗量+40% |

**宠物行为逻辑（统一通过 PetAISystem 驱动）：**
- 每个宠物有一个"跟随偏移量"（相对于召唤师的固定位置，如石头人在前、治愈之灵在后）
- 跟随状态下，宠物 Lerp 到目标偏移位置
- 敌人进入宠物攻击范围 → 宠物锁定该敌人 → 攻击冷却到 → 执行攻击
- 锁定优先级：距离最近 > 有标记 > 正在攻击召唤师
- 攻击完成后回到跟随状态

### 宠物配合 / 连携系统

**核心机制：元素标记 + 双元素触发反应**

每种元素攻击命中敌人时，在敌人身上挂一个**元素标记**（独立计时器，持续 2.5 秒）。

```
// 每个敌人携带的数据结构
{
  marks: [
    { element: 'fire',  expireAt: 4200 },   // 游戏时间戳
    { element: 'ice',   expireAt: 4500 },
  ]
}
```

**元素反应表（两两触发，每次触发消耗参与的两个标记）：**

| 组合 | 名称 | 效果 | 视觉 |
|------|------|------|------|
| 火 🔥 + 冰 ❄️ | **融化** | 目标受到 25 点额外伤害，周围 60px 内敌人减速 40%（3秒） | 蒸汽爆炸粒子 |
| 冰 ❄️ + 雷 ⚡ | **超导** | 目标防御力清零（5秒内受伤害×1.5），且连锁闪电可额外弹跳 1 次 | 蓝色电弧扩散 |
| 火 🔥 + 雷 ⚡ | **过载** | 目标立刻爆炸，对周围 80px 敌人造成 30 点伤害 + 施加"火标记" | 红紫爆炸 |
| 任意元素 + 治愈 💚 | **庇护** | 治愈之灵下次治疗量×2，且治疗溢出量转化为岩石傀儡的护盾 | 金色光环 |

**连携示例 1：火 + 冰 = 融化（最直观的 combo）**
- 火焰精灵火球击中敌人 A → 挂火标记
- 冰霜之狼 0.3 秒后扑咬敌人 A → 挂冰标记
- SynergySystem 检测到 A 身上同时有火+冰 → 触发"融化"
- 结果：A 受到 25 额外伤害（可能直接秒杀小怪），周围敌人被蒸汽减速
- 玩家感受："冰狼配合火精灵能清一大片！"

**连携示例 2：冰 + 雷 = 超导 → 连锁爆发**
- 冰霜之狼冲到怪群中扑咬 → 给 2-3 个敌人挂冰标记
- 雷电之眼释放闪电链 → 命中带冰标记的敌人后弹跳+1 → 7次弹跳覆盖大半屏
- SynergySystem 检测到冰+雷重叠 → 触发"超导"，被弹跳到的敌人防御清零 → 后续弹跳伤害暴增
- 玩家感受："冰狼冲进去挂冰，雷眼清屏！"

**实现要点（SynergySystem）：**
1. 每帧遍历 `allEnemies`，但只检查 `marks.length >= 2` 的（绝大多数敌人没标记，快速跳过）
2. 用两个嵌套 for 检查所有标记对 → 查表触发反应
3. 触发后删除已消耗的标记（防止同一对反复触发）
4. 冷却：同一对敌人同一组合有 1 秒全局冷却（防止帧帧触发）
5. 反应效果由 SynergyDefs.ts 配置表驱动——新增连携只需加一行配置

### 成长升级系统

#### 局内升级 (Roguelike 三选一)

**升级触发：** 击杀敌人 → 掉落经验宝石（绿色小圆） → 玩家走近自动吸取 → XP 条满 → 暂停游戏 → 弹出 3 张卡。

**卡池结构（约 25-30 张卡）：**

| 稀有度 | 权重 | 示例卡片 |
|--------|------|---------|
| 普通 (白) | 55% | 攻击力+12%、移速+8%、拾取范围+20%、宠物HP+15%、弹幕速度+10% |
| 稀有 (蓝) | 30% | 新增宠物（随机未拥有的）、全宠物攻速+20%、经验获取+25%、连携伤害+40% |
| 史诗 (金) | 15% | 宠物数量上限+1、解锁"元素共鸣"（连携触发时全屏伤害）、减伤光环（周围宠物受伤-20%） |

**宠物升级线（示例：火焰精灵）：**
- Lv1 → Lv2: 伤害+25%
- Lv2 → Lv3: 火球变为三连发（扇形 30° 散射）
- Lv3 → Lv4: 爆炸范围+50%，火球颜色变蓝
- Lv4 → Lv5 (MAX): 击中后附加灼烧 DOT（每秒 5 伤害，持续 3 秒，可叠加）

宠物升级通过拿到对应的"强化 X 宠物"卡片实现。初始宠物为 Lv1，每拿到一张对应卡升一级。

#### 局外成长（永久，localStorage）

**货币：** 每局结算获得金币 = `存活秒数 × 1 + 击杀数 × 0.5`

**成长树：**
```
基础攻击力    [0/10]  每级 +5%    初始 → 10 → 30 → 60 → 100... 金币递增
```

### 性能方案

| 策略 | 具体措施 | 预期收益 |
|------|---------|---------|
| **对象池** | 敌人/弹幕/粒子/VFX 全部走 `Phaser.GameObjects.Group` 的 `get()` + `kill()` 模式。预分配：敌人200、弹幕150、粒子300、伤害数字50。 | 消灭 GC 抖动，帧率稳定 |
| **空间网格** | 屏幕划分为 100×100px 格子，每个实体注册到网格。碰撞检测只看 p×q 九宫格内的对。替代遍历 `allEnemies × allProjectiles`。 | 在200敌人+50弹幕场景下，碰撞检测对数量从 10,000 降到~500（20倍） |
| **逻辑与渲染分离** | 伤害/寻路等逻辑在定时器 20Hz 执行，渲染在 60fps requestAnimationFrame。Phaser 的 `update` 中区分 `logicTick` 和 `renderTick`。 | 移动端 CPU 压力减半 |
| **视野裁剪** | 超出屏幕边界 64px 的精灵 `setVisible(false)`，逻辑照跑但不送渲染管线。 | GPU fill rate 限制时救命 |
| **纹理图集** | 所有宠物/敌人/特效精灵打包一个 Texture Atlas（TexturePacker 或免费工具），一次 GPU 上传。 | draw call 从 ~200 降到 < 10 |
| **粒子限制** | 全局粒子计数器，超过上限(300)时最早的粒子被强制回收。低端机(检测到帧率 < 30fps)自动降到150。 | 防止 GPU 带宽耗尽 |
| **简易血量** | 敌人血量 < 3 时用整数而非浮点。大量敌人死亡检查用 `hp <= 0` 而非 `hp < 0.001`。 | 微不足道但零成本 |
| **Service Worker 缓存** | 注册 SW，预缓存 index.html + bundle.js + atlas.png。第二次打开秒开。 | 加载体验从"等3秒"变"秒开" |

**目标帧率：**
- 中端机（骁龙 7 系 / iPhone 11+）：稳定 55-60fps，屏幕 150+ 敌人 + 5 宠物 + 50 弹幕
- 低端机（骁龙 6 系 / iPhone 8）：稳定 30fps，触发自动降级（粒子减半、敌人上限 100）

## 关键决策与取舍

1. **Phaser 3 而非 PixiJS** — PixiJS 渲染快但"裸"。选 Phaser 的核心理由是"AI 生成友好度"：Phaser 有大量现成的 game jam / tutorial 代码在训练集中，AI 生成 Phaser 代码的 bug 率明显低于生成自定义引擎代码。1.2MB bundle 在 SW 缓存后是 0ms，不构成否决理由。

2. **ECS-lite 而非纯 ECS / 纯 OOP** — 纯 ECS（bitecs）对 AI 来说过于抽象，生成的 query/component 组合经常错配。纯 OOP（继承链）在宠物种类多时会膨胀（`StoneGolem extends MeleePet extends BasePet extends PhaserSprite`）。选择折中：每个实体就是一个 Phaser Sprite/Container，带一个 `data` 对象存逻辑属性；System 是纯函数或单例类，遍历实体数组操作。

3. **元素反应连携而非硬编码 combo** — 如果把连携写成"当火焰精灵和冰霜之狼同时攻击同一敌人→触发融化"，代码会是 O(N²) 的宠物对检查。改用元素标记系统后：任意宠物挂标记→系统扫描标记组合→触发反应。新增宠物或连携只需改配置表，零逻辑代码改动。扩展性高了一个数量级。

4. **竖屏而非横屏** — 手机单手操作天然竖屏。横屏需要双手，不适合通勤/排队场景。代价是视野稍窄，但通过合理的摄像机缩放（1×）和宠物跟随偏移（圆形围绕）可以弥补。

5. **全屏拖拽优先，虚拟摇杆备选** — 虚拟摇杆在"割草游戏"中体验欠佳（需要精确对准摇杆区域，战斗中容易脱手）。全屏任意位置拖动方向更直觉，也节省摇杆 UI 占用的屏幕面积。但保留虚拟摇杆作为一个可选设置。

6. **无后端，纯本地** — 范围已明确 out 了商业化/服器/联机。localStorage 足够，且避免了"服务器宕机→demo 不能玩"的演示事故。

## 风险与缓解

### 🔴 P0 — 最可能翻车

**风险 1：200 敌人同屏时帧率崩溃（< 20fps）**
- **原因：** Phaser Arcade Physics 的 `overlap()` 全量检测是 O(N×M)，且每个 overlap 回调触发 JS ↔ native 桥接。移动端 CPU 单核性能不足时直接卡死。
- **缓解：**
  - 第一天就实现 SpatialGrid，不做"先跑起来再优化"——事后加性能层容易改崩。
  - W1 D6 做一次压力测试（SpawnSystem 爆 200 敌人），目标是 30fps+。不达标不进入 W2。
  - 降级预案：将碰撞检测频率降到 15Hz，伤害结算独立于碰撞检测（位置重叠即判定，不需要 arcade body）。
  - 终极降级：放弃 Phaser Arcade Physics，碰撞全都用 SpatialGrid + 手动距离计算。代码量增加 ~200 行但性能有保障。

**风险 2：元素标记系统逻辑 bug 导致连携不触发或无限触发**
- **原因：** 标记过期清理、反应冷却、同一个组合多次触发——这三个边界条件容易写出死循环或静默失效。
- **缓解：**
  - 写 `SynergySystem` 之前，先写单元测试（标记过期、双标记触发、冷却阻挡、同帧不重复触发）
  - 每个敌人存 `lastReactionTime`，同类型反应有 1s 全局冷却
  - 反应触发后立即 `splice` 消耗的标记，不依赖 "expired 标志位" 做延迟清理

**风险 3：宠物 AI 在密集怪群中行为诡异（频繁切换目标/抖动）**
- **原因：** 每帧都在重新评估"最近敌人"，导致宠物在两个挨得很近的敌人之间反复横跳。
- **缓解：**
  - 宠物锁定目标后有"锁定粘性"：只有当前目标死亡 或 离开攻击范围×1.5 时才切换
  - 切换冷却 0.5 秒——短时间内不更换目标
  - 如果当前目标还在范围内且存活 → 不重新寻敌（性能 + 行为稳定性双赢）

**风险 4：localStorage 数据损坏导致局外成长清零**
- **原因：** 写一半浏览器崩溃、用户清理缓存、数据结构升级不兼容。
- **缓解：**
  - `SaveManager.save()` 走 write-then-validate 模式：先写 `_temp` key，验证 JSON parse 成功，再 rename 到正式 key
  - 读时 catch JSON parse error → 回退到默认值 → 不弹报错
  - `version` 字段 + 迁移函数，数据结构变了自动升级

### 🟡 P1 — 需要关注但不致命

**风险 5：Phaser 场景切换时内存泄漏**
- 每次 GameScene restart（玩家重开），旧的 sprite/texture 没释放，多次重开后 OOM。
- **缓解：** GameScene.shutdown() 中显式调用 `this.children.removeAll(true)` + Group 的 `clear(true, true)`。定期做 10 次重开内存测试。

**风险 6：移动端"全屏拖拽"与浏览器手势冲突**
- iOS Safari 的边缘滑动（后退）、下拉刷新、双击缩放 会吞掉触摸事件。
- **缓解：** `<meta name="viewport" content="... user-scalable=no">` + `touch-action: none` on canvas + `e.preventDefault()` on touchmove。iPhone 真机测试 D1 就做。

**风险 7：AI 生成的 TypeScript 代码类型体操过头**
- AI 可能写出过度泛型化的代码（如 `class Entity<T extends ComponentMap>`），导致编译报错 cascade，人看不懂也改不了。
- **缓解：** 在 AGENTS.md / 项目 README 中明确约束："避免过度抽象，优先具体类型。Pet 就是 `Pet` 类型，别搞 `Entity<PetComponent>`"。

### 🟢 P2 — 低概率低影响

**风险 8：jsfxr 音效在 iOS Safari 上无声**
- iOS 的 WebAudio 要求用户交互后才能播放。第一个音效可能在用户触摸前触发。
- **缓解：** TitleScene 中加一个"点击开始"按钮，用户点击时调用 `Howler.ctx.resume()` 或 Phaser 的 `sound.unlock()`。

## 我放弃的备选

- **PixiJS 方案**：渲染性能更好（WebGL 批量渲染比 Phaser 快约 2×），但需要自建物理、场景管理、对象池、输入系统——代码量翻倍，AI 生成质量不稳定。如果后续性能测试发现 Phaser Arcade Physics 确实是瓶颈，可以局部替换：碰撞检测改用纯 SpatialGrid + 距离计算，保留 Phaser 的渲染/输入/场景。
- **Godot 4 + Web 导出**：Godot 的 H5 导出产物 ~15MB，加载时间 10 秒，不适合移动端。且 GDScript/C# 在"AI 生成代码"约束下训练数据远不如 TypeScript + Phaser。
- **纯 Canvas 2D（无框架）**：最轻量（bundle < 50KB），但 WebGL 加速用不上意味着满屏敌人时 CPU 软渲染直接跪。适合作为"终极降级方案"的 Canvas 回退模式（Phaser 已内置），不适合做主渲染。
- **全屏虚拟摇杆（唯一方案）**：固定摇杆在快节奏割草中体验差——拇指滑出摇杆区域就停止移动。全屏拖拽才是正确答案。
- **回合制 / 半即时**：节省性能但丢掉"割草爽感"。这次设计目标就是保留吸幸式爽感，不做阉割。
