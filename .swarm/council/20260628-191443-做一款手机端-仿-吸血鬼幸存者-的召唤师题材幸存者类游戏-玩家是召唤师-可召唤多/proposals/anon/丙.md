## 一句话方案

用 **Phaser 3 + TypeScript + Vite** 做一款竖屏 H5 召唤师幸存者 Demo：玩家只负责走位和构筑宠物队伍，3-5 种宠物自动作战并通过“标记 / 元素 / 阵型”连携放大输出，局内经验升级三选一驱动爽感，中端手机通过对象池、空间网格、Canvas/WebGL 批渲染和伤害节流保证大量怪潮下可玩帧率。

## 推荐方案

### 技术栈与渲染

- **框架**：Phaser 3。
- **语言**：TypeScript。
- **构建**：Vite。
- **渲染方案**：优先 Phaser WebGL renderer，自动 fallback Canvas；素材以几何图形、sprite atlas、免费音效为主。

选择 Phaser 3 的原因：

1. **AI 生成代码友好**：Phaser 3、TypeScript、Vite 的训练语料充足，AI 能稳定生成场景、输入、物理、动画、资源加载、UI 等常见结构。
2. **移动 H5 成熟**：Phaser 对触摸输入、缩放、音频解锁、WebGL 批渲染和浏览器兼容有成熟方案。
3. **原型速度快**：内置 Scene、GameObjects、Tweens、Camera、Input、Loader，几周内做可玩 Demo 比 PixiJS + 自建游戏框架更稳。
4. **性能可控**：不使用 Arcade Physics 做全部单位碰撞，核心战斗采用自写轻量数据层 + 空间网格，Phaser 主要负责渲染和输入，避免大量物理体拖垮手机。

### 整体代码架构

采用 **“轻量 ECS 思路的数据驱动架构”，不引入复杂 ECS 库**。原因是 Demo 期需要清晰、低依赖、AI 易维护：实体用数字 id 和 TypedArray/普通数组存核心状态，系统按固定顺序更新。

关键主循环：

1. `InputSystem`：读取虚拟摇杆向量，更新召唤师移动意图。
2. `SpawnSystem`：按计时器、难度曲线和屏幕边缘刷新怪物。
3. `PetAISystem`：宠物寻敌、跟随主人、施法冷却、阵型调整。
4. `EnemySystem`：敌人朝召唤师推进，处理简单避让或分层速度扰动。
5. `ProjectileSystem`：更新投射物、命中检测、生命周期。
6. `CombatSystem`：处理伤害、状态、连携触发、死亡。
7. `ExpSystem`：经验球掉落、吸附、升级。
8. `RenderSyncSystem`：把数据层位置、血量、状态同步到 Phaser GameObjects。
9. `UISystem`：HUD、暂停、升级三选一、结算。

关键模块清单：

- `GameScene`：主战斗场景，持有系统调度与时间轴。
- `BootScene / PreloadScene`：初始化、加载资源。
- `UIScene`：HUD、选卡、结算浮层，和战斗 Scene 解耦。
- `EntityManager`：管理玩家、宠物、敌人、投射物、经验球 id。
- `ObjectPool<T>`：敌人、子弹、伤害数字、经验球、特效复用。
- `SpatialGrid`：固定网格空间划分，用于敌人查询、AOE、投射物命中。
- `PetConfig / EnemyConfig / UpgradeConfig`：所有数值配置化，便于 AI 后续扩内容。
- `SaveService`：localStorage 读写局外成长、最高存活时间、设置。
- `BalanceService`：难度曲线、刷新权重、升级池权重。

碰撞与空间划分：

- 地图按固定格子切分，例如 `128px × 128px`。
- 敌人每帧写入 `SpatialGrid`，宠物攻击只查询目标附近格子，不全局遍历。
- 近战、AOE、吸经验都用圆形距离平方判断。
- 子弹命中使用“查询当前格和邻格 + 半径判断”，不使用复杂物理引擎。

对象池：

- 敌人、投射物、经验球、短命特效全部池化。
- 敌人死亡只隐藏并回收，不销毁 GameObject。
- 伤害数字可默认关闭或限频显示，低端机可自动降级。

### 3-5 种原型宠物设计

建议首版做 5 种，但 W1 先完成 3 种，W2 补齐：

1. **狼灵**
   - 定位：近战单体 / 前排。
   - 攻击方式：自动扑向最近敌人，短距离撕咬，命中后返回召唤师附近或继续追击。
   - 特色：对“被标记”目标造成额外伤害，承担基础割草手感。

2. **火鸦**
   - 定位：远程穿透 / 点燃。
   - 攻击方式：绕召唤师盘旋，定时发射火羽，火羽可穿透 2-4 个敌人。
   - 特色：给敌人叠加 `Burn`，持续掉血，适合清线。

3. **藤妖**
   - 定位：控制 / AOE 辅助。
   - 攻击方式：在敌群脚下生成藤蔓圈，短暂减速并每 0.5 秒造成伤害。
   - 特色：把敌人聚成低速团，给火鸦和雷灵创造输出窗口。

4. **雷灵**
   - 定位：跳跃链 / 爆发。
   - 攻击方式：对最近敌人释放闪电，随后跳到附近 2-5 个目标。
   - 特色：对处于 `Wet / Rooted / Marked` 的目标增加跳跃次数或暴击。

5. **守护龟**
   - 定位：防御 / 回复 / 容错。
   - 攻击方式：低频震地小 AOE，主要提供护盾光环。
   - 特色：每隔一段时间给召唤师加护盾，升级后可嘲讽小范围敌人或减伤。

### 宠物配合 / 连携机制

推荐核心机制：**“召唤印记 + 元素反应”双层连携，但首版只做 2 个反应，避免复杂失控。**

通用实现：

- 每个敌人有 `statusFlags` 和 `statusStacks`，例如 `Marked`、`Burn`、`Rooted`。
- 每次宠物命中时调用 `ComboSystem.onHit(attackerPetType, targetEnemyId, hitPayload)`。
- `ComboSystem` 根据目标状态和攻击类型触发额外效果，并设置内置冷却，防止同一敌人每帧无限触发。
- 连携伤害使用独立系数，受宠物等级、召唤师天赋和局内升级影响。

具体连携例子 1：**狼灵 + 火鸦：猎印爆燃**

- 触发：
  - 狼灵攻击有 35% 概率给敌人挂 `Marked`，持续 4 秒。
  - 火鸦火羽命中 `Marked` 敌人时触发“爆燃”。
- 计算：
  - 爆燃造成 `火鸦伤害 × 0.8 + 狼灵等级 × 4` 的小范围 AOE。
  - AOE 半径 `80px`，最多影响 8 个敌人。
  - 每个被标记敌人每 1.2 秒最多触发一次，避免性能和数值爆炸。
- 体验：
  - 玩家感觉不是自己攻击，而是“狼灵锁定猎物，火鸦引爆猎物”，突出指挥召唤主题。

具体连携例子 2：**藤妖 + 雷灵：缚雷连锁**

- 触发：
  - 藤妖藤蔓圈内敌人获得 `Rooted` 或 `Slowed`。
  - 雷灵命中 `Rooted/Slowed` 敌人时，链式闪电额外跳跃 2 次。
- 计算：
  - 基础跳跃 `3` 次；若首目标有 `Rooted/Slowed`，变为 `5` 次。
  - 每次跳跃伤害衰减 15%。
  - 每次跳跃只查询目标所在格周边 `160px` 内最近敌人。
- 体验：
  - 藤妖把怪潮“定住成团”，雷灵负责“清屏跳电”，形成非常直观的组合构筑。

### 成长升级系统

局内基础循环：

1. 怪物死亡掉经验球。
2. 召唤师靠近吸取，或升级后获得更大吸附半径。
3. 经验条满时暂停战斗，弹出“升级三选一”。
4. 玩家选择一张卡，立即强化宠物、召唤师或连携。
5. 继续战斗，怪潮按时间增强。

升级三选一卡池示例：

- 宠物获得类：
  - `召唤狼灵`：若未拥有则新增狼灵，已拥有则狼灵等级 +1。
  - `召唤火鸦`：新增或升级火鸦。
  - `召唤藤妖`：新增或升级藤妖。
  - `召唤雷灵`：新增或升级雷灵。
  - `召唤守护龟`：新增或升级守护龟。
- 宠物强化类：
  - `兽群号令`：所有宠物攻击速度 +12%。
  - `灵魂契约`：宠物伤害 +15%，召唤师最大生命 -5%，做风险收益。
  - `迅捷召回`：宠物离主人过远时回位速度 +30%。
- 连携强化类：
  - `猎印扩散`：猎印爆燃命中 3 个以上敌人时，有 25% 概率给附近敌人扩散 `Marked`。
  - `缚雷增幅`：雷灵命中减速敌人时额外跳跃 +1。
  - `元素共鸣`：触发任意连携后，所有宠物冷却缩短 0.2 秒，内置冷却 3 秒。
- 召唤师生存类：
  - `灵能护盾`：每 20 秒获得一层护盾。
  - `轻身术`：移动速度 +8%。
  - `拾魂范围`：经验吸附范围 +25%。

宠物等级提升：

- 每种宠物等级 1-8。
- 升级方向固定但可配置，例如：
  - 等级提升增加基础伤害、冷却、范围、投射物数量。
  - 关键等级解锁质变：火鸦 4 级穿透 +1，藤妖 5 级藤蔓圈 +1，狼灵 6 级标记概率提升。
- 卡池权重优先给已拥有宠物，保证玩家能形成构筑，而不是随机太散。

可选局外成长雏形：

- 只做轻量 localStorage，不做账号和服务器。
- 结算后获得“魂石”，可点 3 个永久升级：
  - 初始生命 +5%。
  - 初始移动速度 +3%。
  - 初始随机宠物等级 +1，最多 1 级。
- 局外成长必须弱，不破坏 Demo 平衡，主要用于展示可扩展方向。

### 移动端操作方案与 UI 布局

操作：

- 竖屏优先，推荐设计分辨率 `720 × 1280`，用 Phaser Scale `FIT + CENTER_BOTH`。
- 左下角虚拟摇杆：
  - 手指按下出现摇杆底盘。
  - 拖拽控制移动方向和速度。
  - 松手停止。
- 右下角保留一个“召唤爆发 / 集火指令”按钮作为可选增强：
  - 首版可以做成“所有宠物优先攻击最近精英 / Boss，持续 4 秒”。
  - 若赶工，可只展示按钮但不做复杂技能；核心仍是自动战斗。

HUD：

- 顶部中间：存活时间、当前难度阶段。
- 顶部左侧：生命 / 护盾。
- 顶部右侧：暂停按钮、性能降级提示。
- 底部：经验条。
- 屏幕右侧小图标列：当前宠物和等级。
- 升级弹窗：三张大卡片纵向或横向排列，手机上卡片要足够大，显示名称、图标、效果和“新召唤 / 升级 / 连携”标签。
- 结算 UI：存活时间、击杀数、最高等级、触发连携次数、获得魂石、重新开始按钮。

### 性能方案

目标：中端手机浏览器在 150-300 个敌人、数十个投射物和少量特效时保持接近 30-60 FPS，优先保证可玩。

具体策略：

1. **不把每个敌人做成完整 Phaser 物理体**
   - 敌人移动和碰撞由自写数据系统处理。
   - Phaser 只渲染 sprite 或简单形状。

2. **固定空间网格**
   - 敌人每帧按格子登记。
   - 宠物寻敌、AOE、子弹命中只查附近格。
   - 避免 `宠物 × 敌人` 或 `子弹 × 敌人` 全量遍历。

3. **对象池**
   - 敌人、子弹、经验球、爆炸特效、伤害数字全部复用。
   - 每波刷新从池中取对象，死亡时回池。

4. **伤害结算节流**
   - AOE 地面效果每 `200ms` 或 `500ms` tick 一次，不每帧伤害。
   - 连携触发有 per-target cooldown。
   - DOT 每 `500ms` tick。

5. **批量渲染和素材控制**
   - 同类敌人使用同一 texture / atlas。
   - 几何占位阶段使用少量 texture，不频繁动态绘图。
   - 关闭昂贵滤镜、动态光照和复杂粒子。

6. **敌人行为简化**
   - 普通敌人只朝召唤师移动，加少量随机速度偏移。
   - 不做复杂寻路。
   - 敌人之间不做真实碰撞，只做极简分离或干脆允许重叠，用视觉分层解决。

7. **动态降级**
   - FPS 连续 3 秒低于 28 时：
     - 降低伤害数字数量。
     - 降低粒子数量。
     - 降低最大同时敌人软上限。
     - 延长部分视觉特效刷新间隔。

## 关键决策与取舍（选 X 而非 Y，因为…）

1. **选 Phaser 3 而非 PixiJS**
   - PixiJS 渲染更纯粹，但需要自建更多游戏层能力；Phaser 对场景、输入、资源、音频、移动适配更完整，几周 Demo 更稳。

2. **选 TypeScript 而非 JavaScript**
   - 宠物、敌人、升级、状态、连携配置会快速膨胀，TypeScript 能减少 AI 迭代时的字段拼错和接口漂移。

3. **选轻量 ECS 思路而非引入完整 ECS 库**
   - 完整 ECS 学习和调试成本高，AI 生成代码也容易过度抽象；用数组 + 系统模块能获得主要性能收益，同时保持可读。

4. **选自动战斗 + 走位 + 构筑，不做主动普攻**
   - 题材核心是召唤师“指挥而非亲砍”，主动攻击会削弱差异化；玩家操作负担也更适合手机。

5. **选 2 个明确连携而非做复杂元素体系**
   - 连携是硬核心，但 Demo 期复杂元素反应容易数值爆炸和难解释；两个强反馈连携足够证明方向。

6. **选 localStorage 轻局外成长而非账号 / 云存档**
   - 符合零成本和范围约束，能展示长期成长雏形，但不引入服务器复杂度。

7. **选竖屏虚拟摇杆而非横屏双摇杆**
   - 竖屏更符合 H5 传播和单手试玩；游戏不需要瞄准摇杆，左手拖拽移动即可。

## 步骤 / 任务分解（按“几周”给出里程碑切分，如 W1/W2/W3 每周交付物）

交付物：

- 实现召唤师移动、虚拟摇杆、基础 HUD。
- 实现敌人对象池、刷新系统、难度随时间递增。

验收重点：

- 玩家能通过选卡组成至少 2 种有效构筑。

- 增加 5-10 张升级卡。

验收重点：

- 首次打开 15 秒内理解操作和目标。

## 风险与缓解

1. **大量敌人导致手机掉帧**
   - 缓解：从 W1 开始就用对象池和数据层移动；W2 加空间网格；W3 做动态降级。不要先用 Arcade Physics 堆满敌人再重构。

2. **连携机制看不懂**
   - 缓解：只保留 2 个首发连携，配强视觉标签和升级卡描述；结算展示“触发连携次数”，强化玩家认知。

3. **升级卡太随机，构筑不成型**
   - 缓解：卡池权重优先已有宠物；前 3 次升级提高“新召唤宠物”出现概率；同一宠物升级路径稳定。

4. **召唤师本人太弱导致无聊或无力**
   - 缓解：给召唤师足够移动速度、拾取成长、护盾和“集火指令”按钮；玩家虽然不攻击，但通过走位、选卡、指挥影响战局。

5. **AI 生成代码后期难维护**
   - 缓解：强制配置驱动，模块边界清晰，避免全塞进一个 Scene；所有宠物和升级以 `Config` 表声明。

6. **移动浏览器音频和触摸兼容问题**
   - 缓解：首次触摸后解锁音频；所有按钮触摸热区大于 44px；禁止页面滚动；适配安全区。

7. **数值失控，连携秒屏或完全没用**
   - 缓解：连携设置内置冷却、目标数量上限、伤害系数表；每周固定做 30 分钟数值试玩记录。

## 成本 / 工时估计（诚实标出最贵/最易翻车的部分）

按 1 名熟悉前端的开发者 + 顶级 AI 辅助估算：

- W1 最小闭环：4-6 个工作日。
- W2 连携和成长：5-7 个工作日。
- W3 性能和移动端打磨：4-6 个工作日。
- W4 发布包装：2-4 个工作日。

总计：

- **可玩原型**：约 2-3 周。
- **可外发 Demo**：约 3-4 周。
- **现金成本**：约 0，使用开源框架、免费占位素材、AI 生成图标和音效，部署用免费静态托管。

最贵 / 最易翻车部分：

1. **性能优化**：如果前期用错架构，后期重构成本最大；必须从第一周就避免全量碰撞和大量对象创建。
2. **移动端手感**：虚拟摇杆、触摸、屏幕密度、UI 热区需要真机调试，桌面模拟不可靠。
3. **数值平衡**：割草游戏的爽感高度依赖成长曲线，不能只靠功能完成，需要反复试玩。
4. **连携可读性**：机制有趣但不显眼会等于没做，需要视觉、音效、文案共同支撑。

## 验收标准 (done-when，可检验)

基础可玩：

- 玩家用手机浏览器打开后，不需要键盘鼠标即可开始游戏。
- 召唤师可通过虚拟摇杆移动，松手停止，触摸区域不误触页面滚动。
- 怪物从屏幕外持续刷新并追向召唤师，刷新强度随时间增加。
- 玩家死亡后进入结算页，可重新开始。

召唤师三核心：

- 至少有 5 种宠物配置，其中至少 3 种在首局可实际获得。
- 每种宠物都能自动寻敌并作战，不需要玩家瞄准。
- 玩家本体没有强主动普攻，主要输出来自宠物。
- 至少 2 个宠物连携可实际触发，并有视觉或 UI 反馈。
- 每种宠物至少有 1-8 级成长配置，升级后战斗表现有可感知变化。

吸幸式循环：

- 怪物死亡掉经验球。
- 拾取经验能升级。
- 升级时游戏暂停并出现三选一卡。
- 选择卡牌后强化立即生效。
- 10 分钟内至少出现 15 次以上升级选择，节奏不长时间断档。

性能：

- 中端 Android Chrome 真机或等价设备上，150 个敌人同屏时仍能稳定操作。
- 300 个敌人压力场景下不崩溃，允许降级到 30 FPS 左右。
- 运行 10 分钟后内存不持续明显上涨，没有大量未回收 GameObject。

存档：

- localStorage 能保存最高存活时间、魂石、局外成长。
- 刷新页面后数据仍存在。
- 存档损坏或不存在时能安全回退默认值。

UI / 结算：

- HUD 显示生命、经验、时间、宠物等级。
- 升级卡文字在手机屏幕上可读，按钮足够大。
- 结算页显示存活时间、击杀数、等级、连携次数、获得资源。

可维护：

- 宠物、敌人、升级卡、连携参数均在配置文件中可调。
- 新增一种宠物不需要改主循环，只需增加配置和一个可插拔行为模块。
- 核心系统有最少 smoke test 或调试开关：刷怪压力测试、升级卡测试、连携触发测试。

## 我放弃的备选（一段话，便于主席之后做混合方案）

我放弃了 Unity WebGL、Godot Web、纯 Canvas 自研和 PixiJS 自建框架：Unity/Godot 的 Web 移动包体、加载和浏览器兼容风险偏高，不适合零成本几周 H5 原型；纯 Canvas 虽然极轻，但输入、资源、场景、音频和 UI 都要自建，AI 后续迭代更容易散；PixiJS 性能很好，但游戏框架层需要补太多。也放弃了复杂职业、装备、地图事件、肉鸽房间、账号云存档和商业化，因为它们会稀释三核心，首版应集中证明“召唤多宠 + 宠物连携 + 升级割草”是否好玩。
