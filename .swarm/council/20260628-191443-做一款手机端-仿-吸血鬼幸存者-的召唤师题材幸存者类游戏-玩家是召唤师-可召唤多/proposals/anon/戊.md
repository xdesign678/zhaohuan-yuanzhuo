# 召唤师幸存者 H5 — DeepSeek 方案

## 一句话方案

**Phaser 3 + TypeScript + Vite**，ECS-lite 架构，5种宠物 + 元素反应连携 + 三选一升级卡 + 虚拟摇杆，3周交付可玩 Demo。

---

## 推荐方案

### 框架与语言

| 维度 | 选择 | 理由 |
|------|------|------|
| 游戏框架 | **Phaser 3.80+** | 内置 Arcade 物理（碰撞检测零成本）、场景管理（标题→游戏→结算天然切换）、对象池（Group 自带）、音频、输入、补间动画——"电池全满"。在"AI 生成代码"场景下，Phaser 3 的社区教程量是 PixiJS 的 5-10 倍，意味着 AI 训练语料丰富得多，生成代码准确性更高。 |
| 语言 | **TypeScript (strict)** | 约束是"AI 持续迭代"，TypeScript 的类型系统就是 AI 的护栏——编译期就能抓到"宠物数组里塞了个敌人"这种低级错误。 |
| 构建 | **Vite 5.x** | HMR 极快，零配置启动，打包产物小（tree-shaking）。配合 `vite-plugin-compression` 产出 gzip，首屏加载 < 1MB。 |
| 渲染 | **Phaser AUTO (WebGL 优先, Canvas 回退)** | 现代手机浏览器 WebGL 覆盖率 > 97%。Phaser 的 WebGL 渲染管线已经做过大量移动端调优，内部自动批量合并同纹理 draw call。 |
| 美术 | **几何图形 + 免费 sprite sheet (Kenney/itch.io)** | 原型阶段：圆形+颜色区分就够了。宠物 = 带方向箭头的彩色圆，敌人 = 深色骷髅圆。素材加载尺寸 < 200KB。 |
| 音效 | **jsfxr (程序化生成 8-bit 音效)** | 攻击/受伤/升级/连携触发都有对应音效，零网络加载，生成代码即可，体积 ~5KB。 |

**为什么不选 PixiJS？** PixiJS 渲染更快（2x），但那是裸渲染——它没有物理、没有场景、没有对象池。在"AI 生成代码"的约束下，选 PixiJS 意味着 AI 要凭空造一套游戏引擎出来，代码量 ×2~3，bug 密度 ×3，交付时间 ×2。Phaser 的 1.2MB bundle 在 Service Worker 缓存后是 0ms 加载，不值得用开发效率去换。

### 整体架构

**设计原则：不走纯 ECS，走"Phaser GameObject + Data Component"的 ECS-lite。**

原因：纯 ECS（如 bitecs/geotic）在 AI 代码生成时容易出"entity 里没挂对应 component 就 query 到了"这种鬼 bug。Phaser 自带的 Sprite/Group 本身就是 entity 容器，在它上面挂一个 `data` 对象（`sprite.setData('unitData', {...})`）是最自然的方式。

```
src/
```

**主循环 (GameScene.update):**
```
1. VirtualJoystick → Player.velocity
2. Player.update() — 移动 + 经验磁铁
3. PetAISystem.update() — 每个宠物寻敌 → 攻击 → 生成弹幕
4. SynergySystem.update() — 扫描所有敌人的元素标记 → 触发连携
5. SpawnSystem.update() — 按波次曲目表出怪
6. Enemy.update() × N — AI 寻路向玩家移动
7. Phaser Arcade Physics overlap/collide — 弹幕命中检测 (走 SpatialGrid 过滤后)
8. DamageSystem.update() — 结算伤害 + 死亡回收
9. XPSystem.update() — 经验宝石拾取 → 升级触发 → 弹出 UpgradePanel
10. WaveSystem.update() — 计时器 + 难度递增
11. HUD.update() — 刷新 UI
```

### 5种原型宠物设计

| # | 名称 | 攻击方式 | 定位 | 攻击范围 | 攻击间隔 | 基础伤害 | 特殊机制 |
|---|------|---------|------|---------|---------|---------|---------|
| 1 | **岩石傀儡** 🪨 | 近战冲撞，撞击最近敌人 | 前排坦克 | 60px | 2.0s | 15 | 攻击附带小范围击退；每10秒自动嘲讽周围敌人（吸引仇恨） |
| 2 | **火焰精灵** 🔥 | 发射火球，命中产生60px爆炸 | 远程AOE | 250px | 1.2s | 20 | 爆炸溅射伤害 = 直接伤害的50%；给目标上"火标记" |
| 3 | **冰霜之狼** ❄️ | 快速扑咬，三连击 | 近战控制 | 80px | 0.6s (×3) | 8×3 | 每击附加"冰标记"；身上已有冰标记的敌人被再次击中 → 冰冻（定身1秒） |
| 4 | **雷电之眼** ⚡ | 闪电链，弹跳至附近敌人 | 远程连锁 | 200px | 1.5s | 18 | 最多弹跳4次，每次伤害衰减15%；弹跳优先找"有元素标记"的敌人；给目标上"雷标记" |
| 5 | **治愈之灵** 💚 | 每2.5秒对召唤师+血量最低宠物释放治疗 | 辅助治疗 | 150px | 2.5s | 治疗12 | 不攻击；治疗时若目标身上有元素标记，治疗量+40% |

**宠物行为逻辑（统一通过 PetAISystem 驱动）：**
- 每个宠物有一个"跟随偏移量"（相对于召唤师的固定位置，如石头人在前、治愈之灵在后）
- 跟随状态下，宠物 Lerp 到目标偏移位置
- 敌人进入宠物攻击范围 → 宠物锁定该敌人 → 攻击冷却到 → 执行攻击
- 锁定优先级：距离最近 > 有标记 > 正在攻击召唤师
- 攻击完成后回到跟随状态

### 宠物配合 / 连携系统

**核心机制：元素标记 + 双元素触发反应**

每种元素攻击命中敌人时，在敌人身上挂一个**元素标记**（独立计时器，持续 2.5 秒）。

```
// 每个敌人携带的数据结构
{
  marks: [
    { element: 'fire',  expireAt: 4200 },   // 游戏时间戳
    { element: 'ice',   expireAt: 4500 },
  ]
}
```

**元素反应表（两两触发，每次触发消耗参与的两个标记）：**

| 组合 | 名称 | 效果 | 视觉 |
|------|------|------|------|
| 火 🔥 + 冰 ❄️ | **融化** | 目标受到 25 点额外伤害，周围 60px 内敌人减速 40%（3秒） | 蒸汽爆炸粒子 |
| 冰 ❄️ + 雷 ⚡ | **超导** | 目标防御力清零（5秒内受伤害×1.5），且连锁闪电可额外弹跳 1 次 | 蓝色电弧扩散 |
| 火 🔥 + 雷 ⚡ | **过载** | 目标立刻爆炸，对周围 80px 敌人造成 30 点伤害 + 施加"火标记" | 红紫爆炸 |
| 任意元素 + 治愈 💚 | **庇护** | 治愈之灵下次治疗量×2，且治疗溢出量转化为岩石傀儡的护盾 | 金色光环 |

**连携示例 1：火 + 冰 = 融化（最直观的 combo）**
- 火焰精灵火球击中敌人 A → 挂火标记
- 冰霜之狼 0.3 秒后扑咬敌人 A → 挂冰标记
- SynergySystem 检测到 A 身上同时有火+冰 → 触发"融化"
- 结果：A 受到 25 额外伤害（可能直接秒杀小怪），周围敌人被蒸汽减速
- 玩家感受："冰狼配合火精灵能清一大片！"

**连携示例 2：冰 + 雷 = 超导 → 连锁爆发**
- 冰霜之狼冲到怪群中扑咬 → 给 2-3 个敌人挂冰标记
- 雷电之眼释放闪电链 → 命中带冰标记的敌人后弹跳+1 → 7次弹跳覆盖大半屏
- SynergySystem 检测到冰+雷重叠 → 触发"超导"，被弹跳到的敌人防御清零 → 后续弹跳伤害暴增
- 玩家感受："冰狼冲进去挂冰，雷眼清屏！"

**实现要点（SynergySystem）：**
1. 每帧遍历 `allEnemies`，但只检查 `marks.length >= 2` 的（绝大多数敌人没标记，快速跳过）
2. 用两个嵌套 for 检查所有标记对 → 查表触发反应
3. 触发后删除已消耗的标记（防止同一对反复触发）
4. 冷却：同一对敌人同一组合有 1 秒全局冷却（防止帧帧触发）
5. 反应效果由 SynergyDefs.ts 配置表驱动——新增连携只需加一行配置

### 成长升级系统

#### 局内升级 (Roguelike 三选一)

**升级触发：** 击杀敌人 → 掉落经验宝石（绿色小圆） → 玩家走近自动吸取 → XP 条满 → 暂停游戏 → 弹出 3 张卡。

**卡池结构（约 25-30 张卡）：**

| 稀有度 | 权重 | 示例卡片 |
|--------|------|---------|
| 普通 (白) | 55% | 攻击力+12%、移速+8%、拾取范围+20%、宠物HP+15%、弹幕速度+10% |
| 稀有 (蓝) | 30% | 新增宠物（随机未拥有的）、全宠物攻速+20%、经验获取+25%、连携伤害+40% |
| 史诗 (金) | 15% | 宠物数量上限+1、解锁"元素共鸣"（连携触发时全屏伤害）、减伤光环（周围宠物受伤-20%） |

**宠物升级线（示例：火焰精灵）：**
- Lv1 → Lv2: 伤害+25%
- Lv2 → Lv3: 火球变为三连发（扇形 30° 散射）
- Lv3 → Lv4: 爆炸范围+50%，火球颜色变蓝
- Lv4 → Lv5 (MAX): 击中后附加灼烧 DOT（每秒 5 伤害，持续 3 秒，可叠加）

宠物升级通过拿到对应的"强化 X 宠物"卡片实现。初始宠物为 Lv1，每拿到一张对应卡升一级。

#### 局外成长（永久，localStorage）

**货币：** 每局结算获得金币 = `存活秒数 × 1 + 击杀数 × 0.5`

**成长树：**
```
基础攻击力    [0/10]  每级 +5%    初始 → 10 → 30 → 60 → 100... 金币递增
基础 HP       [0/10]  每级 +8%
移速          [0/5]   每级 +4%
经验倍率      [0/5]   每级 +10%
起始宠物等级  [0/3]   每级: 开局随机一只宠物从 Lv2 开始
拾取范围      [0/5]   每级 +10%
```

存储结构：
```typescript
interface SaveData {
  coins: number;
  upgrades: Record<string, number>;  // "atk" → 3, "hp" → 1, ...
  version: number;                    // 版本号，用于数据迁移
  stats: { totalKills: number; bestTime: number; gamesPlayed: number };
}
```

### 移动端操作与UI布局

**竖屏优先设计（基准分辨率 390×844 iPhone 14 逻辑像素，等比缩放 + safe area 适配）。**

**操作：左侧固定虚拟摇杆**
- 位置：屏幕左下角，距左 20px，距底 80px（给宠物状态栏留空间）
- 大小：外圈半径 45px，内圈（拇指实际拖动区）半径 25px
- 实现：`pointerdown` 在摇杆区域外 → 不响应。在区域内 → 记录起始点 → `pointermove` 计算偏移向量 → 钳制在半径内 → 映射为玩家速度方向
- 备选方案（可从设置切换）：**全屏滑哪走哪**——任何位置触摸拖动，玩家朝拖动方向移动。对"割草游戏"来说更直观，不遮挡视野。两个方案都实现，默认全屏拖拽。

**HUD 布局：**

**升级选卡面板（暂停 overlay）：**
- 半透明黑色遮罩（rgba(0,0,0,0.7)）覆盖全屏
- 3 张卡片竖排居中，卡片间距 12px
- 卡片：白底圆角矩形（160×200px），顶部图标（64×64），中间名称，底部效果描述
- 稀有度用边框颜色区分（白/蓝/金）
- 点击卡片 → 震动反馈（navigator.vibrate(15)）→ 应用升级 → 关闭面板 → 继续

### 性能方案

| 策略 | 具体措施 | 预期收益 |
|------|---------|---------|
| **对象池** | 敌人/弹幕/粒子/VFX 全部走 `Phaser.GameObjects.Group` 的 `get()` + `kill()` 模式。预分配：敌人200、弹幕150、粒子300、伤害数字50。 | 消灭 GC 抖动，帧率稳定 |
| **空间网格** | 屏幕划分为 100×100px 格子，每个实体注册到网格。碰撞检测只看 p×q 九宫格内的对。替代遍历 `allEnemies × allProjectiles`。 | 在200敌人+50弹幕场景下，碰撞检测对数量从 10,000 降到~500（20倍） |
| **逻辑与渲染分离** | 伤害/寻路等逻辑在定时器 20Hz 执行，渲染在 60fps requestAnimationFrame。Phaser 的 `update` 中区分 `logicTick` 和 `renderTick`。 | 移动端 CPU 压力减半 |
| **视野裁剪** | 超出屏幕边界 64px 的精灵 `setVisible(false)`，逻辑照跑但不送渲染管线。 | GPU fill rate 限制时救命 |
| **纹理图集** | 所有宠物/敌人/特效精灵打包一个 Texture Atlas（TexturePacker 或免费工具），一次 GPU 上传。 | draw call 从 ~200 降到 < 10 |
| **粒子限制** | 全局粒子计数器，超过上限(300)时最早的粒子被强制回收。低端机(检测到帧率 < 30fps)自动降到150。 | 防止 GPU 带宽耗尽 |
| **简易血量** | 敌人血量 < 3 时用整数而非浮点。大量敌人死亡检查用 `hp <= 0` 而非 `hp < 0.001`。 | 微不足道但零成本 |
| **Service Worker 缓存** | 注册 SW，预缓存 index.html + bundle.js + atlas.png。第二次打开秒开。 | 加载体验从"等3秒"变"秒开" |

**目标帧率：**
- 中端机（骁龙 7 系 / iPhone 11+）：稳定 55-60fps，屏幕 150+ 敌人 + 5 宠物 + 50 弹幕
- 低端机（骁龙 6 系 / iPhone 8）：稳定 30fps，触发自动降级（粒子减半、敌人上限 100）

## 关键决策与取舍

1. **Phaser 3 而非 PixiJS** — PixiJS 渲染快但"裸"。选 Phaser 的核心理由是"AI 生成友好度"：Phaser 有大量现成的 game jam / tutorial 代码在训练集中，AI 生成 Phaser 代码的 bug 率明显低于生成自定义引擎代码。1.2MB bundle 在 SW 缓存后是 0ms，不构成否决理由。

2. **ECS-lite 而非纯 ECS / 纯 OOP** — 纯 ECS（bitecs）对 AI 来说过于抽象，生成的 query/component 组合经常错配。纯 OOP（继承链）在宠物种类多时会膨胀（`StoneGolem extends MeleePet extends BasePet extends PhaserSprite`）。选择折中：每个实体就是一个 Phaser Sprite/Container，带一个 `data` 对象存逻辑属性；System 是纯函数或单例类，遍历实体数组操作。

3. **元素反应连携而非硬编码 combo** — 如果把连携写成"当火焰精灵和冰霜之狼同时攻击同一敌人→触发融化"，代码会是 O(N²) 的宠物对检查。改用元素标记系统后：任意宠物挂标记→系统扫描标记组合→触发反应。新增宠物或连携只需改配置表，零逻辑代码改动。扩展性高了一个数量级。

4. **竖屏而非横屏** — 手机单手操作天然竖屏。横屏需要双手，不适合通勤/排队场景。代价是视野稍窄，但通过合理的摄像机缩放（1×）和宠物跟随偏移（圆形围绕）可以弥补。

5. **全屏拖拽优先，虚拟摇杆备选** — 虚拟摇杆在"割草游戏"中体验欠佳（需要精确对准摇杆区域，战斗中容易脱手）。全屏任意位置拖动方向更直觉，也节省摇杆 UI 占用的屏幕面积。但保留虚拟摇杆作为一个可选设置。

6. **无后端，纯本地** — 范围已明确 out 了商业化/服器/联机。localStorage 足够，且避免了"服务器宕机→demo 不能玩"的演示事故。

## 风险与缓解

### 🔴 P0 — 最可能翻车

**风险 1：200 敌人同屏时帧率崩溃（< 20fps）**
- **原因：** Phaser Arcade Physics 的 `overlap()` 全量检测是 O(N×M)，且每个 overlap 回调触发 JS ↔ native 桥接。移动端 CPU 单核性能不足时直接卡死。
- **缓解：**
  - 第一天就实现 SpatialGrid，不做"先跑起来再优化"——事后加性能层容易改崩。
  - W1 D6 做一次压力测试（SpawnSystem 爆 200 敌人），目标是 30fps+。不达标不进入 W2。
  - 降级预案：将碰撞检测频率降到 15Hz，伤害结算独立于碰撞检测（位置重叠即判定，不需要 arcade body）。
  - 终极降级：放弃 Phaser Arcade Physics，碰撞全都用 SpatialGrid + 手动距离计算。代码量增加 ~200 行但性能有保障。

**风险 2：元素标记系统逻辑 bug 导致连携不触发或无限触发**
- **原因：** 标记过期清理、反应冷却、同一个组合多次触发——这三个边界条件容易写出死循环或静默失效。
- **缓解：**
  - 写 `SynergySystem` 之前，先写单元测试（标记过期、双标记触发、冷却阻挡、同帧不重复触发）
  - 每个敌人存 `lastReactionTime`，同类型反应有 1s 全局冷却
  - 反应触发后立即 `splice` 消耗的标记，不依赖 "expired 标志位" 做延迟清理

**风险 3：宠物 AI 在密集怪群中行为诡异（频繁切换目标/抖动）**
- **原因：** 每帧都在重新评估"最近敌人"，导致宠物在两个挨得很近的敌人之间反复横跳。
- **缓解：**
  - 宠物锁定目标后有"锁定粘性"：只有当前目标死亡 或 离开攻击范围×1.5 时才切换
  - 切换冷却 0.5 秒——短时间内不更换目标
  - 如果当前目标还在范围内且存活 → 不重新寻敌（性能 + 行为稳定性双赢）

**风险 4：localStorage 数据损坏导致局外成长清零**
- **原因：** 写一半浏览器崩溃、用户清理缓存、数据结构升级不兼容。
- **缓解：**
  - `SaveManager.save()` 走 write-then-validate 模式：先写 `_temp` key，验证 JSON parse 成功，再 rename 到正式 key
  - 读时 catch JSON parse error → 回退到默认值 → 不弹报错
  - `version` 字段 + 迁移函数，数据结构变了自动升级

### 🟡 P1 — 需要关注但不致命

**风险 5：Phaser 场景切换时内存泄漏**
- 每次 GameScene restart（玩家重开），旧的 sprite/texture 没释放，多次重开后 OOM。
- **缓解：** GameScene.shutdown() 中显式调用 `this.children.removeAll(true)` + Group 的 `clear(true, true)`。定期做 10 次重开内存测试。

**风险 6：移动端"全屏拖拽"与浏览器手势冲突**
- iOS Safari 的边缘滑动（后退）、下拉刷新、双击缩放 会吞掉触摸事件。
- **缓解：** `<meta name="viewport" content="... user-scalable=no">` + `touch-action: none` on canvas + `e.preventDefault()` on touchmove。iPhone 真机测试 D1 就做。

**风险 7：AI 生成的 TypeScript 代码类型体操过头**
- AI 可能写出过度泛型化的代码（如 `class Entity<T extends ComponentMap>`），导致编译报错 cascade，人看不懂也改不了。
- **缓解：** 在 AGENTS.md / 项目 README 中明确约束："避免过度抽象，优先具体类型。Pet 就是 `Pet` 类型，别搞 `Entity<PetComponent>`"。

### 🟢 P2 — 低概率低影响

**风险 8：jsfxr 音效在 iOS Safari 上无声**
- iOS 的 WebAudio 要求用户交互后才能播放。第一个音效可能在用户触摸前触发。
- **缓解：** TitleScene 中加一个"点击开始"按钮，用户点击时调用 `Howler.ctx.resume()` 或 Phaser 的 `sound.unlock()`。

## 我放弃的备选

- **PixiJS 方案**：渲染性能更好（WebGL 批量渲染比 Phaser 快约 2×），但需要自建物理、场景管理、对象池、输入系统——代码量翻倍，AI 生成质量不稳定。如果后续性能测试发现 Phaser Arcade Physics 确实是瓶颈，可以局部替换：碰撞检测改用纯 SpatialGrid + 距离计算，保留 Phaser 的渲染/输入/场景。
- **Godot 4 + Web 导出**：Godot 的 H5 导出产物 ~15MB，加载时间 10 秒，不适合移动端。且 GDScript/C# 在"AI 生成代码"约束下训练数据远不如 TypeScript + Phaser。
- **纯 Canvas 2D（无框架）**：最轻量（bundle < 50KB），但 WebGL 加速用不上意味着满屏敌人时 CPU 软渲染直接跪。适合作为"终极降级方案"的 Canvas 回退模式（Phaser 已内置），不适合做主渲染。
- **全屏虚拟摇杆（唯一方案）**：固定摇杆在快节奏割草中体验差——拇指滑出摇杆区域就停止移动。全屏拖拽才是正确答案。
- **回合制 / 半即时**：节省性能但丢掉"割草爽感"。这次设计目标就是保留吸幸式爽感，不做阉割。
